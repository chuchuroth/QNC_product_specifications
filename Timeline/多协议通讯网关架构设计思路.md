
# 多协议通讯网关架构设计思路

以下是针对**多协议机器人通讯网关**的三种架构模式的完整设计思路。每种模式都从**数据流、内部模型、任务划分、关键实现、适用场景**五个维度展开。

---

# 方案一：协议转换器 (Protocol Converter)

## 核心理念
> **A口进，B口出**。网关作为两个（或多个）异构网络之间的翻译官，每个方向有明确的转换规则。

## 1. 数据流架构

```
┌─────────┐      ┌─────────────┐      ┌─────────┐
│ CANopen │◄────│  协议转换器  │────►│PROFINET│
│  网络   │      │   (网关)    │      │  网络   │
└─────────┘      └──────┬──────┘      └─────────┘
                        │
                   ┌────▼────┐
                   │ 可选的  │
                   │ 第三网络│
                   └─────────┘
```

**典型流向**：
- **正向转换**：CANopen PDO → 内部格式 → PROFINET IO 数据
- **反向转换**：PROFINET 控制字 → 内部格式 → CANopen SDO/PDO
- **旁路转发**：某些信号直接透传（如急停）

## 2. 内部数据模型（轻量级）

**不维护完整过程映像**，只维护**转换映射表**：

```c
typedef struct {
    uint32_t source_protocol;   // 0=CANopen, 1=PROFINET, 2=Modbus
    uint32_t source_addr;       // CANopen: 索引+子索引; PROFINET: slot+subslot
    uint32_t target_protocol;
    uint32_t target_addr;
    uint8_t  direction;          // 0=单向, 1=双向
    uint8_t  scaling_type;       // 线性/查表/自定义函数
    float    scale_factor;
    float    offset;
    uint32_t deadband;           // 变化超过此值才转发
    uint32_t last_value;
    uint32_t last_timestamp;
} ConversionRule;
```

**特点**：
- 没有统一的对象字典，只有一张转换规则表（通常几十到几百条）
- 规则在启动时从配置文件加载，运行时只增删改（极少）
- 每个规则独立处理，互不干扰

## 3. 任务划分

| 任务名 | 优先级 | 周期 | 职责 |
| :--- | :--- | :--- | :--- |
| **CANopen 接收** | 高 (3) | 1ms | 接收PDO，放入原始数据队列 |
| **PROFINET 接收** | 高 (3) | 1ms | 接收IO数据，放入原始数据队列 |
| **转换引擎** | 中 (2) | 2ms | 轮询队列，查表执行转换，产生输出指令 |
| **CANopen 发送** | 中 (2) | 异步 | 发送SDO/PDO到CAN网络 |
| **PROFINET 发送** | 中 (2) | 周期性 | 更新IO输出缓冲区 |
| **配置/诊断** | 低 (0) | 100ms | 处理规则配置、输出诊断信息 |

## 4. 关键实现细节

### 4.1 转换规则存储
```
配置文件 (JSON/XML/二进制)
    ↓ 启动时加载
内存中的转换规则数组 (静态分配，最大N条)
    ↓ 运行时CRC校验
快速查找哈希表 (key = source_protocol + source_addr)
```

### 4.2 数据一致性保证
```c
// 使用双缓冲 + 版本号
struct {
    ConversionRule rules[MAX_RULES];
    uint32_t version;      // 每次修改递增
    uint32_t crc32;
} rule_set_active, rule_set_shadow;

// 修改规则时
memcpy(&rule_set_shadow, &rule_set_active, sizeof);
// 修改 rule_set_shadow.rules[...]
rule_set_shadow.version++;
rule_set_shadow.crc32 = calc_crc(&rule_set_shadow);
// 原子切换指针
active_rules = &rule_set_shadow;
```

### 4.3 死区/滤波机制
```c
bool should_forward(uint32_t new_value, ConversionRule* rule) {
    uint32_t delta = abs((int32_t)new_value - (int32_t)rule->last_value);
    if (delta > rule->deadband) {
        rule->last_value = new_value;
        return true;
    }
    return false;  // 变化太小，不转发
}
```

## 5. 适用场景

| 场景 | 是否推荐 | 理由 |
| :--- | :--- | :--- |
| 连接两个明确的异构网络（如CANopen伺服 → PROFINET PLC） | ✅ 最佳 | 转换规则清晰，数据量可控 |
| 协议网关产品（如Anybus、EtherCAT-Junction） | ✅ 典型 | 市场成熟，有现成方案参考 |
| 机器人控制器内部多个子系统互连 | ⚠️ 一般 | 规则数可能膨胀到上千条，管理复杂 |
| 需要任意设备间相互通信 | ❌ 不适合 | 缺乏灵活性，每增加一对设备需新增规则 |

# 方案二：数据总线 (Data Bus)

## 核心理念
> **发布-订阅**。所有协议栈平等地读写一个共享的“过程映像”，网关只是总线的载体。

## 1. 数据流架构

```
                    ┌─────────────────────────────────┐
                    │        过程映像 (Process Image) │
                    │  ┌─────┐ ┌─────┐ ┌─────┐        │
                    │  │点1  │ │点2  │ │点3  │ ...    │
                    │  └──▲──┘ └──▲──┘ └──▲──┘        │
                    │     │       │       │           │
                    └─────┼───────┼───────┼───────────┘
                          │       │       │
    ┌─────────┐     ┌─────┴───────┴───────┴─────┐     ┌─────────┐
    │ CANopen │     │       数据总线核心        │     │PROFINET│
    │  设备   │◄───►│  • 读写仲裁              │◄───►│   PLC   │
    └─────────┘     │  • QoS管理               │     └─────────┘
    ┌─────────┐     │  • 时间戳/质量戳         │     ┌─────────┐
    │ Modbus  │◄───►│  • 持久化(可选)          │◄───►│EtherNet/│
    │  设备   │     └──────────────────────────┘     │   IP    │
    └─────────┘                                      └─────────┘
```

## 2. 内部数据模型（完整过程映像）

```c
// 数据点描述符
typedef struct {
    uint32_t id;                    // 唯一标识符 (hash from name)
    char     name[64];              // "robot.joint1.position"
    uint8_t  type;                  // INT8/UINT16/FLOAT32/STRING...
    uint8_t  access;                // RO/RW/RW_EXCLUSIVE
    uint32_t size;                  // 字节数
    void*    storage;               // 指向实际存储位置
    uint32_t timestamp_ns;          // 最后更新时间 (单调时钟)
    uint8_t  quality;               // 0=BAD, 1=UNCERTAIN, 2=GOOD
    uint8_t  protocol_source;       // 哪个协议写的
    uint32_t subscriber_mask;       // 哪些协议订阅了这个点
} DataPoint;

// 过程映像管理
typedef struct {
    DataPoint*   points;            // 数据点数组
    uint32_t     point_count;
    SemaphoreHandle_t rw_lock;      // 读写锁（支持多读单写）
    uint32_t     global_version;    // 整个映像版本号
} ProcessImage;
```

## 3. 任务划分

| 任务名 | 优先级 | 周期 | 职责 |
| :--- | :--- | :--- | :--- |
| **CAN发布者** | 高 (3) | 1ms | 读CAN PDO → 写过程映像 |
| **CAN订阅者** | 中 (2) | 2ms | 读过程映像 → 发送CAN PDO |
| **PROFINET发布者** | 高 (3) | 1ms | 读PROFINET IO → 写过程映像 |
| **PROFINET订阅者** | 中 (2) | 2ms | 读过程映像 → 更新PROFINET输出 |
| **QoS监视器** | 中 (1) | 10ms | 检查每个点的更新频率，标记STALE |
| **持久化任务** | 低 (0) | 1000ms | 将关键点保存到Flash（可选） |
| **诊断任务** | 低 (0) | 100ms | 发布诊断信息点 |

**关键设计**：发布者和订阅者是**分离的任务**，这样允许不同协议有不同的读写周期。



# 方案三：混合模式 (Hybrid)

## 核心理念
> **关键路径走总线，非关键/边界场景走转换**。根据数据类型动态选择最优路径。

## 1. 数据流架构

```
                              ┌─────────────────────────────────────┐
                              │           路由决策引擎              │
                              │  ┌──────────┐  ┌──────────────┐    │
                              │  │规则引擎  │  │动态优先级   │    │
                              │  │(静态配置)│  │(运行时学习) │    │
                              │  └────┬─────┘  └──────┬───────┘    │
                              └───────┼────────────────┼───────────┘
                                      │                │
                    ┌─────────────────┼────────────────┼─────────────────┐
                    │                 │                │                 │
                    ▼                 ▼                ▼                 ▼
            ┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────┐
            │ 实时总线    │    │ 高速数据流  │    │ 配置/诊断   │    │ 异常    │
            │ (过程映像)  │    │ (零拷贝通道)│    │ (转换器)    │    │ 旁路    │
            └──────┬──────┘    └──────┬──────┘    └──────┬──────┘    └────┬────┘
                   │                  │                  │                │
                   └──────────────────┼──────────────────┼────────────────┘
                                      │                  │
                              ┌───────▼──────────────────▼───────┐
                              │         协议栈适配层             │
                              │  CAN │ PROFINET │ Modbus │ ETH/IP │
                              └──────────────────────────────────┘
```

**三通道并行**：
- **通道1（实时总线）**：周期性控制数据（位置、速度、扭矩）→ 走过程映像 + QoS监控
- **通道2（高速直通）**：海量数据（点云、图像）→ 走零拷贝共享内存，不经过过程映像
- **通道3（配置/诊断）**：非周期性参数 → 走协议转换器模式，延迟不敏感

## 2. 内部数据模型（分层）

```c
// 第1层：实时过程映像（小数据，高频率）
typedef struct {
    // 预定义的关键点，编译时确定，内存连续
    struct {
        int32_t  joint_positions[6];
        int32_t  joint_torques[6];
        uint16_t digital_inputs;
        uint16_t digital_outputs;
        uint8_t  robot_state;      // 0=IDLE,1=RUN,2=ERROR...
    } realtime_map;
    uint32_t version;
    uint64_t timestamp_ns;
} RealtimeProcessImage;

// 第2层：高速数据流通道（大数据，零拷贝）
typedef struct {
    void*     buffer;              // 预分配的环形缓冲区
    uint32_t  buffer_size;
    uint32_t  write_index;
    uint32_t  read_index;
    uint8_t   stream_id;           // 0=激光雷达, 1=相机, 2=力传感器...
    uint32_t  frame_counter;
} HighSpeedStream;

// 第3层：配置数据库（低频率，可转换）
// 使用方案一的 ConversionRule 数组
```

## 3. 任务划分（优先级精细化管理）

| 优先级 | 任务 | 周期 | 职责 | 路径 |
| :--- | :--- | :--- | :--- | :--- |
| **最高(7)** | 高速流DMA | 中断 | 触发高速数据搬移 | 通道2直通 |
| **高(6)** | 实时总线-发布者 | 500μs | 更新实时过程映像 | 通道1 |
| **高(5)** | 实时总线-订阅者 | 500μs | 从实时映像发送指令 | 通道1 |
| **中(4)** | 路由决策 | 1ms | 判断每个数据包走哪条路 | 决策层 |
| **中(3)** | 转换引擎 | 5ms | 处理配置/诊断数据转换 | 通道3 |
| **低(1)** | 诊断/日志 | 100ms | 汇总状态，输出日志 | 通道3 |
| **最低(0)** | 配置持久化 | 1000ms | 保存配置到Flash | 通道3 |

## 4. 关键实现细节

### 4.1 路由决策规则引擎
```c
// 静态规则表 (编译时可配置)
typedef struct {
    uint32_t signal_id;          // 如 "joint1_position"
    uint8_t  data_size;          // 字节数
    uint8_t  max_latency_us;     // 最大允许延迟
    uint8_t  min_period_us;      // 最小周期
    uint8_t  route_type;         // 0=实时映像, 1=高速流, 2=转换器
    uint8_t  fallback_route;     // 超限时的备选路径
} RoutingRule;

// 动态优先级调整 (运行时学习)
void routing_decision_maker(void* arg) {
    while(1) {
        // 监控每个通道的当前负载和延迟
        uint32_t bus_load = get_bus_utilization();
        uint32_t avg_latency = get_avg_latency_us();
        
        // 动态调整路由
        for each signal {
            if (signal.current_latency > signal.max_allowed_latency) {
                // 升级到更快通道
                if (signal.route != FASTEST_ROUTE) {
                    promote_signal_to_faster_channel(signal);
                }
            } else if (signal.current_latency < signal.min_latency && 
                       bus_load > 80) {
                // 延迟充裕，降级到较慢通道以释放资源
                demote_signal_to_slower_channel(signal);
            }
        }
        
        vTaskDelay(pdMS_TO_TICKS(10));
    }
}
```

### 4.2 高速流零拷贝实现
```c
// 生产者 (如网卡驱动接收到点云数据)
uint32_t produce_high_speed_frame(uint8_t stream_id, void* data, uint32_t len) {
    HighSpeedStream* stream = &streams[stream_id];
    
    // 检查是否有足够空间
    uint32_t available = (stream->read_index - stream->write_index - 1) % stream->buffer_size;
    if (available < len) {
        return ERR_BUFFER_FULL;
    }
    
    // 直接写入环形缓冲区 (无锁，使用原子操作)
    uint32_t pos = stream->write_index;
    memcpy(&stream->buffer[pos], data, len);
    
    // 发布写入完成 (原子更新)
    atomic_store(&stream->write_index, (pos + len) % stream->buffer_size);
    
    // 通知消费者 (可选，如果消费者在等待)
    xSemaphoreGive(stream->data_ready_sem);
    
    return len;
}

// 消费者 (如另一个协议栈要转发此数据)
void* consume_high_speed_frame(uint8_t stream_id, uint32_t* len) {
    HighSpeedStream* stream = &streams[stream_id];
    
    // 等待新数据 (带超时)
    if (xSemaphoreTake(stream->data_ready_sem, pdMS_TO_TICKS(10)) == pdTRUE) {
        uint32_t read_pos = atomic_load(&stream->read_index);
        uint32_t write_pos = atomic_load(&stream->write_index);
        
        if (read_pos != write_pos) {
            *len = (write_pos - read_pos) % stream->buffer_size;
            void* data = &stream->buffer[read_pos];
            
            // 注意：不拷贝，直接返回指针！调用方必须及时处理
            // 并在使用完后 atomic_update(&stream->read_index, ...)
            return data;
        }
    }
    return NULL;
}
```

### 4.3 异常旁路（当首选路径失败时）
```c
// 监控实时映像的健康状态
void health_monitor_task(void* arg) {
    uint32_t last_good_version = 0;
    uint32_t stale_counter = 0;
    
    while(1) {
        uint32_t current_version = realtime_image.version;
        if (current_version == last_good_version) {
            stale_counter++;
            if (stale_counter > 10) {  // 10ms无更新
                // 触发旁路：将关键信号切换到转换器模式
                activate_fallback_for_critical_signals();
            }
        } else {
            stale_counter = 0;
            last_good_version = current_version;
        }
        
        vTaskDelay(pdMS_TO_TICKS(1));
    }
}

// 旁路动作：将控制指令通过SDO/Modbus直接写入设备
void activate_fallback_for_critical_signals() {
    // 告诉路由决策引擎，临时走通道3（转换器模式）
    routing_table[CTRLWORD_SIGNAL].route_type = 2;  // 强制转换器
    routing_table[TARGET_POS_SIGNAL].route_type = 2;
    
    // 启动看门狗，5秒后尝试恢复
    start_recovery_timer(5000);
}
```

## 5. 适用场景

| 场景 | 是否推荐 | 理由 |
| :--- | :--- | :--- |
| 复杂机器人系统（机械臂+视觉+力控+安全PLC） | ✅ 最佳 | 不同数据自然分流，发挥各自优势 |
| 需要兼顾实时控制和数据采集 | ✅ 推荐 | 实时控制走总线，采集走转换器 |
| 研发阶段的探索性项目 | ⚠️ 过度设计 | 初期复杂度太高，先走总线模式 |
| 产品需要适应多种未知场景 | ✅ 推荐 | 灵活性最高，可根据配置切换模式 |

# 三种方案对比总结

| 维度 | 协议转换器 | 数据总线 | 混合模式 |
| :--- | :--- | :--- | :--- |
| **核心数据流** | 点对点映射 | 发布-订阅 | 动态路由 |
| **数据模型复杂度** | 低（映射表） | 高（完整映像） | 极高（分层+规则） |
| **代码规模** | 5k-15k行 | 15k-30k行 | 30k-60k行 |
| **RAM需求** | 256KB-512KB | 1MB-4MB | 2MB-8MB |
| **实时性（最小延迟）** | 100-500μs | 200-1000μs | 50-500μs（可优化） |
| **确定性** | 高（转换延迟固定） | 中（受锁竞争影响） | 高（关键路径固定） |
| **扩展性** | 差（每增加设备要加规则） | 好（只需写适配器） | 极好（自动路由） |
| **调试难度** | 低 | 中 | 高 |
| **开发周期** | 2-4个月 | 4-8个月 | 8-12个月 |
| **成熟度** | 极成熟（无数网关产品） | 成熟（类似ROS） | 前沿（自定义） |

---

## 最终建议：分阶段演进路线

```
阶段1 (1-2月)
    └── 实现协议转换器的最小原型
        ├── CANopen ↔ 内部表格
        ├── Modbus ↔ 内部表格
        └── 验证基础转发正确性
                │
                ▼
阶段2 (3-4月)
    └── 升级为数据总线模式
        ├── 建立统一过程映像
        ├── 实现QoS和质量戳
        ├── 添加第三个协议
        └── 压力测试 (1000点, 1kHz)
                │
                ▼
阶段3 (5-8月)
    └── 可选：增加混合模式特性
        ├── 高速流通道 (用于视觉/点云)
        ├── 动态路由规则
        ├── 异常旁路
        └── 性能调优
```

**如果只能选一个方案起步，我建议从方案二（数据总线）开始**，原因是：
- 协议转换器后期扩展规则爆炸，重构成本高
- 数据总线的架构一旦搭好，增加新协议只是写适配器
- 混合模式可以基于数据总线逐步增加，不需要推翻重来

根据项目具体情况，比如大约需要连接多少种协议、多少设备、数据量多大、实时性要求多高，根据这些参数来判断哪个方案最适合qnc。
