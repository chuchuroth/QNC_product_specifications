# QNC 运维与恢复指南
**文档编号:** QNC-ORG-001
**版本:** 草稿 1.0
**状态:** 针对运维、部署与恢复的派生配套指南
**分类:** 内部 / 项目受控

## 0. 目的与范围

本指南描述了 QNC 在部署、配置、监控、故障处理、重启以及安全模式运行方面的操作规程。它既适用于基线部署,也适用于经批准的扩展功能,尤其是基于 FastDDS 的机器人集成场景,且不会改变源文档中所定义的系统边界。QNC 仍然**未通过安全认证**;安全模式是一项操作性与故障管理功能,并不能替代功能安全。

## 0.1 运维原则

以下五项核心原则在运维过程中为强制要求:
1. **禁止部分激活工件**
2. **仅可激活经过验证且兼容的配置文件、模式与 QoS 工件**
3. **错误必须按状态域进行隔离**
4. **严重错误必须在 1 秒内切换至安全模式**
5. **变更失败时,须立即回滚至上一个已知良好状态**

# 1. 配置操作规程

## 1.1 配置工件与职责划分

QNC 的运维配置必须划分为独立的、经版本控制的工件。系统级的网络、安全及接口设置属于**系统配置(System Configuration)**。命令、参数、遥测、故障及生命周期序列等设备专属映射属于**设备配置文件(Device Profile)**。DDS 专属的类型定义与 QoS **不属于**设备配置文件的边界范围,而是通过 **IDL 类型注册表** 与 **QoS 配置目录** 进行管理。运维过程中不得混淆这些边界。

### 建议的工件结构

| 工件 | 用途 | 归属方 | 激活规则 |
|-----------------------|-------------------------------------------------------------------------|--------------------------------|------------------------------------------------------|
| `system-config`(系统配置) | 网络、安全、已启用接口、生命周期策略 | 运维 / 平台 | 仅支持原子化激活 |
| `device-profile`(设备配置文件) | 设备类别、协议绑定、命令、参数、遥测、故障映射、生命周期序列 | 系统 / 设备集成 | 仅在通过模式、完整性及兼容性检查后方可激活 |
| `dds-idl-registry`(DDS IDL 注册表) | 主题类型与版本 | 机器人技术 / 中间件 | 仅限已批准的类型 |
| `dds-qos-catalog`(DDS QoS 目录) | 按通信类别划分的 QoS 配置 | 机器人技术 / 中间件 / 运维 | 仅限已批准的 QoS 配置 |
| `audit-policy`(审计策略) | 日志记录、保留期限、变更日志 | 运维 / 安全 | 强制性工件 |

上述划分直接源自 PRS、DPS、ICD 及 PSM。

## 1.2 初始试运行分步规程

### 第一步:检查发布状态
每次安装之前,须确认所部署的协议、桥接及 DDS 功能均属于已批准的发布层级。基线协议为 IO-Link、Modbus RTU、EtherNet/IP 适配器及离散数字 IO;FastDDS、MQTT、OPC UA 及其他协议仅可在经批准的扩展中使用。

### 第二步:确立硬件与网络前提条件
在进行配置分配之前,必须先明确供电、布线、端口分配、IP 方案、VLAN 规则,以及对北向与南向网段的访问权限。对于 DDS,还需额外确认目标网段是否允许组播,或是否需要采用发现服务器(Discovery Server)拓扑结构。

### 第三步:加载系统配置
将网络参数、TLS/令牌策略、已启用服务、安全模式策略、日志/审计设置及允许使用的接口,准备为一个系统工件。该变更必须作为单一事务应用;禁止部分激活。

### 第四步:分配设备配置文件
为每台现场设备分配一个身份标识及匹配策略相符的、已批准的 YAML 配置文件。该配置文件必须声明恰好一个设备类别、定义南向绑定关系,并包含所有必需的命令、参数限值、遥测及故障映射。

### 第五步:单独注册 DDS 工件
若启用了 FastDDS,则需单独注册域 ID(Domain ID)、参与者名称(Participant Name)、发现模式、允许使用的接口、IDL 类型及 QoS 配置。DDS 工件不得隐藏或重复出现在设备配置文件内部。

### 第六步:执行激活前验证
在提交之前,至少须成功完成模式验证、签名/完整性检查、兼容性检查、配置文件与协议的核对,以及安全模式/故障映射验证。存在缺陷的工件将被丢弃;系统将保持在上一个已知良好状态。

### 第七步:原子化激活
配置变更仅以原子化方式激活。在切换过程中,不得出现新旧配置混合可见的状态。若激活过程的任何部分失败,将自动触发回滚。

### 第八步:初始化与设备启动
激活成功后,QNC 生命周期序列将依次从 `BOOTING`(启动中)经过 `INITIALIZING`(初始化中)运行至 `READY`(就绪)及 `ACTIVE`(运行中)。在配置文件层面,将执行 `initialization_sequence`(初始化序列),如适用还将执行 `enable_sequence`(启用序列)。

### 第九步:功能验证
初始试运行结束时,必须验证遥测、故障/事件通道、命令接受情况、状态消息、安全模式抑制功能以及审计日志记录。对于 DDS 部署,还需额外增加发现机制、QoS 及主题兼容性检查。

## 1.3 参数定义与建议配置域

### 系统级别
- 网络配置:静态或 DHCP,如需要可配置 VLAN
- 安全:TLS 1.2 及以上版本、基于令牌的身份验证、基于角色的访问控制(RBAC)
- 审计/日志:JSON 格式日志、轮转、保留期限
- 生命周期策略:恢复模式为手动/自动、是否允许维护模式(Service Mode)
- 安全模式策略:强制锁定执行动作,DDS 可选择仅支持"只订阅"模式

### 配置文件相关
- `device_identity`(设备身份)
- 带超时/重试类别的 `protocol_binding`(协议绑定)
- `capabilities`(能力)
- `commands`(命令)
- 带最小值/最大值、类型、单位、比例系数的 `parameters`(参数)
- `telemetry`(遥测)
- 包含严重程度、锁存状态、`safe_mode_required`(是否需要安全模式)的 `faults`(故障)
- 包含 `recover_sequence`(恢复序列)与 `shutdown_sequence`(关闭序列)的 `lifecycle_sequences`(生命周期序列)

### DDS 相关
- `domain_id`(域 ID)
- `participant_name`(参与者名称)
- `discovery_mode`(发现模式,取值为 `SIMPLE` 或 `DISCOVERY_SERVER`)
- `allowed_interface_list`(允许接口列表)
- `ignore_nonapproved_peers`(忽略未批准的对等方)
- `lease_duration_ms`(租约时长,毫秒)
- `announcement_period_ms`(公告周期,毫秒)
- 主题类型版本及所分配的 QoS 配置

## 1.4 验证与核实

| 验证步骤 | 目标 | 失败时的结果 |
|------------------------------|----------------------------------------|---------------------------------------|
| 模式验证 | 结构正确 | 阻止激活 |
| 签名/哈希校验 | 确保完整性 | 拒绝该工件,记录安全故障日志 |
| 兼容性检查 | 固件、配置文件、DDS 版本相匹配 | 拒绝激活 |
| 参数范围检查 | 无不允许的设定值 | 拒绝该命令或配置文件 |
| 安全模式检查 | 严重故障已正确映射 | 阻止发布 |
| 通信检查 | 链路/会话/发现状态稳定 | 保持在 `READY`(就绪)或 `DEGRADED`(降级)状态 |
| 审计检查 | 变更可追溯 | 停止发布流程 |

这些检查项源自 PRS/DPS/ICD/PSM 中要求的治理与符合性规则。

# 2. DDS 域配置指南

## 2.1 DDS 架构原则

DDS 是 QNC 中一项经批准的扩展功能,由其自身的 **DDS 参与者服务** 负责运行。域配置的管理独立于设备配置文件。DDS 相关错误必须被检测并隔离为 **DDS 域故障**;这些错误不得传播至南向服务。

## 2.2 建议的域 ID 策略

源文档要求对域 ID(Domain ID)的分配进行受控管理,但未规定具体的编号策略。因此建议采用以下**运维标准**:

| 范围 | 用途 | 规则 |
|------------|----------------------------------|---------------------------------------------------|
| 0–31 | 实验室 / 开发环境 | 不可用于生产环境 |
| 32–95 | 单一单元 / 独立部署 | 一个单元对应一个域 |
| 96–159 | 生产线 / 区域 | 一条生产线对应一个域 |
| 160–191 | 集成 / 验收测试 | 限时使用 |
| 192–223 | 灾备 / 恢复 / 应急备用 | 仅在受控条件下激活 |
| 224–232 | 保留给中央中间件服务使用 | 仅限平台批准 |
| 233–255 | 锁定 / 特殊情况 | 未经架构批准不得使用 |

该划分方式是为实现扩展性、隔离性以及避免计划外冲突而制定的派生运维策略;由于域 ID 被视为受治理的配置项,而非设备配置文件的一部分,因此该划分方式与文档集保持兼容。

## 2.3 选择发现模式

对于支持组播且参与者数量有限的小型扁平网段,**SIMPLE 发现方式**为默认选项。对于分段的、较大规模的或跨网络的部署场景 —— 尤其是当组播受限或需要更严格的参与者控制时 —— 应采用 **DISCOVERY_SERVER(发现服务器)**方式。基线方案为 SIMPLE;发现服务器被视为一种高级部署选项。

## 2.4 参与者配置

每个参与者都被分配一个稳定的名称、一个明确的域 ID、一项已定义的发现策略,以及一个受限的网络视图。建议使用 `allowed_interface_list`(允许接口列表)并忽略未经批准的对等方,以减少发现过程中的干扰噪声,并仅允许经批准的通信路径。

## 2.5 QoS 标准

QoS 配置按数据类别进行规范性分配:

| QoS 配置 | 使用场景 |
|---------------------|-----------------------------------------------|
| `QOS_CMD_STRICT`(命令严格型) | 命令请求/响应 |
| `QOS_STATUS_FAST`(状态快速型) | 生命周期与状态消息 |
| `QOS_TELEM_STREAM`(遥测流式) | 高频率遥测数据 |
| `QOS_TELEM_CRITICAL`(遥测关键型) | 关键健康数据 |
| `QOS_FAULT_EVENT`(故障事件型) | 故障/告警,对后加入者(late joiners)相关 |
| `QOS_CONFIG_AUDIT`(配置审计型) | 配置与审计事件 |
| `QOS_DIAG_BULK`(诊断批量型) | 诊断/追踪 |

供需(Offer/Request)不匹配的情况不得被静默接受,而应记录为结构化的 DDS 故障。

## 2.6 网络与互操作规则

DDS 主题必须按功能组织(例如命令入口、状态出口、故障事件、诊断)。归一化数据应当**生成一次、多次发布**,而非针对每个消费者重复计算。每个域中仅可激活经过验证的 IDL 版本及已批准的 QoS 组合。

## 2.7 安全模式与 DDS

在安全模式下,若已获批准,DDS 至多可以以**仅订阅**方式保持活动状态。在此状态下,任何通过 DDS 发布或消费的执行动作都必须被阻止。为便于诊断及协调恢复,状态、故障及健康相关流量可保持活动状态。

### DDS 参考架构图

```text
          +-----------------------------+
          |         北向边缘服务          |
          | REST / WS / 日志 / 桥接      |
          +-------------+---------------+
                        |
          +-------------v---------------+
          |         集成与控制层          |
          | 命令代理 / 故障管理器          |
          | 生命周期 / 状态聚合            |
          +-------------+---------------+
                        |
      +-----------------+------------------+
      |                                    |
+-----v----------------+      +------------v-----------+
| 配置文件与语义归一化   |      |    DDS 参与者服务       |
| 配置文件 / 映射        |      | 域 / 发现 / QoS /       |
|                       |      | IDL 验证                |
+-----+----------------+      +------------+-----------+
      |
+-----v-------------------------------+
|         物理与协议适配层             |
| IO-Link / Modbus / EIP / 离散 IO    |
+-------------------------------------+
```

该划分方式与四层架构以及配置文件与 DDS 工件相互分离的治理机制直接对应。

# 3. 部署模式选择标准

## 3.1 可用部署模式

| 模式 | 说明 | 适用场景 |
|-----------------------------|------------------------------------------------------------|-----------------------------------------------------|
| 独立设备网关(Standalone Device Gateway) | 单台 QNC,配备基线协议及标准北向接口 | 单一单元、试运行、较小规模的机器 |
| 分布式边缘节点(Distributed Edge Node) | 多个协议服务,可选桥接 | 产线运行、分段部署、负载分布 |
| 边缘 + DDS 对等节点(Edge + DDS Peer) | QNC 主动参与机器人 DDS 域 | ROS 2 / FastDDS 集成 |
| 云连接边缘(Cloud-Connected Edge) | 通过已批准的服务向更高层级转发数据 | 遥测、车队监控、远程诊断 |
| 高级多网络 DDS(Advanced Multi-Network DDS) | 采用发现服务器的大型环境 | 分段式工厂、多网络环境 |

这些模式源自 PSM 与 PRS。

## 3.2 选择标准

### 独立设备网关
当优先考虑低复杂度、本地控制、设备数量较少以及对中间件依赖度低时,首选此模式。该模式可最大限度减少故障接口并简化恢复流程。

### 分布式边缘节点
当需要协议多样性、分段部署、子功能更高的可用性,以及设备域的组织性分离时,适用此模式。此模式下故障隔离效果更佳,但配置管理要求更高。

### 边缘 + DDS 对等节点
仅当机器人工作负载确实需要真正的 DDS 参与(即基于主题的数据交换、标准化的 IDL 类型以及协调一致的 QoS 配置)时,方可选择此模式。其运维收益在于互操作性;代价则是更高的发现、QoS 及版本管理工作量。

### 云连接边缘
适用于可观测性、历史数据存档、集中式诊断及运营数据分析场景。该模式不应作为硬性本地实时决策的优先选项。

### 高级多网络 DDS
仅在域分段、发现服务器控制及大量参与者均属强制要求,且获得架构批准的情况下方可部署。

## 3.3 权衡矩阵

| 标准 | 独立式 | 分布式 | 边缘 + DDS 对等 | 云连接式 | 高级 DDS |
|----------------------------|------------|-------------|-----------------|-----------------|--------------|
| 运维工作量 | 低 | 中 | 高 | 中 | 极高 |
| 可扩展性 | 低-中 | 高 | 高 | 高 | 极高 |
| 故障隔离 | 中 | 高 | 高(需良好 DDS 治理) | 中 | 高 |
| 资源消耗 | 低 | 中 | 中-高 | 中 | 高 |
| 恢复复杂度 | 低 | 中 | 高 | 中 | 高 |
| 机器人互操作性 | 低 | 中 | 极高 | 低 | 极高 |

该矩阵是基于已记录的模式及架构原则得出的派生型选型辅助工具。

# 4. 故障恢复规程

## 4.1 错误类别

QNC 按来源与状态域对错误进行区分:**生命周期(Lifecycle)**、**连接(Connection)**、**协议(Protocol)**、**配置文件(Profile)**、**设备类别(Device Class)**、**厂商专属(Vendor-Specific)**;此外还包括配置类、安全类及 DDS 域相关故障。这种区分在运维上是强制性的,因为恢复过程必须始终以域为单位分别开始。

## 4.2 通用恢复方案

每个恢复过程均遵循以下顺序:

1. **检测错误**
2. **对错误进行分类并确定严重程度**
3. **若属严重情况,锁定执行动作**
4. **隔离受影响域**
5. **诊断根本原因**
6. **执行有限的恢复动作**
7. **验证状态与遥测数据**
8. **明确恢复至正常运行状态**
9. **完成审计记录与事后复盘**

## 4.3 基于场景的处置规程

### A. 与现场设备的通信中断

**检测:** 链路/协议状态发生变化、遥测数据停止、心跳或轮询失败。严重通信错误必须被及时检测并记录。

**诊断:**
- 检查线缆、端口、电源
- 检查协议专属的会话状态
- 确认当前生效的配置文件是否仍然有效
- 检查设备是否报告了厂商专属故障

**恢复:**
- 启动有限次数的重试/重连序列
- 若配置文件中存在,则执行 `recover_sequence`(恢复序列)
- 若执行动作无法安全继续:强制进入安全模式
- 重新连接后,先稳定状态/健康信息,再释放执行动作

### B. 配置文件错误或配置文件不兼容

**检测:** 模式错误、签名错误、固件/协议版本不兼容,或当前生效配置文件失效。

**恢复:**
- 立即中止新的激活操作
- 恢复至上一个有效的配置文件
- 重新初始化相关会话
- 若错误是在执行动作过程中检测到的:锁存安全模式,并要求执行明确的恢复步骤

### C. 配置错误 / 更新失败

**检测:** 激活错误、验证错误、服务/固件更新失败或违反兼容性要求。

**恢复:**
- 原子化回滚至上一个已知良好状态
- 生成审计与故障记录
- 重新检查配置状态
- 在完成功能测试之前,系统保持在 `READY`(就绪)状态

### D. DDS 发现丢失 / QoS 不匹配 / IDL 不兼容

**检测:** 参与者未激活、发现机制不稳定、供需(offered/requested)不匹配、模式版本不兼容。

**恢复:**
- 隔离 DDS 域;不干扰南向运行
- 检查参与者配置、域 ID、发现模式及 QoS 分配情况
- 允许受影响的端点重新加入
- 若反复失败,回退至预定义的最小功能集(例如仅发布状态/故障信息)
- 仅在成功重新验证后,方可重新启用完整的 DDS 功能

### E. 未授权命令 / 参数超出限值

**检测:** 命令代理或配置文件验证拒绝该请求。

**恢复:**
- 拒绝单条命令;不改变系统状态
- 向调用方返回错误原因
- 将反复出现的违规情况归类为配置或集成问题
- 除非配置文件或故障映射明确要求,否则不自动切换至安全模式

## 4.4 重启策略

| 重启类型 | 使用场景 | 影响 |
|---------------------------|-----------------------------------------|----------------------------------------------|
| 服务重启(Service Restart) | 单个服务出现故障 | 影响最小 |
| 协议适配器重启(Protocol Adapter Restart) | 南向会话出现故障 | 仅影响相关设备类别/实例 |
| 参与者重启(Participant Restart) | 本地 DDS 故障 | DDS 重新加入,南向保持活动状态 |
| 受控系统重启(Controlled System Restart) | 持久化/完整性问题 | 完整的生命周期重启 |
| 回滚 + 重启(Rollback + Restart) | 更新失败后 | 恢复至上一个有效版本 |

该顺序体现了"优先采取最小可行恢复措施"的原则。

## 4.5 状态恢复

重启后,必须恢复配置以及持久化的故障/审计上下文。只有在配置文件、协议绑定以及可选的 DDS 工件均验证通过后,系统方可从 `READY`(就绪)状态转换至 `ACTIVE`(运行中)状态。

# 5. 安全模式运维

## 5.1 触发条件

安全模式(也称为**操作抑制模式**)由以下情况触发:严重通信错误、完整性/配置错误、主动控制过程中配置文件失效、某些与设备或序列相关的故障状况,以及若被配置为严重级别的特定 DDS 域故障。在配置文件层面,这通过故障映射及 `safe_mode_required`(是否需要安全模式)字段定义。

## 5.2 安全模式下的行为

在安全模式下,**新的执行动作命令将被强制拒绝**。状态、健康、故障及诊断查询仍被允许;若已获批准,DDS 可继续以仅订阅模式运行。安全模式是一项风险降低功能,但**并非经认证的安全功能**。

## 5.3 进入规程

1. 锁存严重故障
2. 立即阻止新的执行动作
3. 发布故障/事件及当前系统模式
4. 隔离受影响域
5. 保持诊断路径畅通
6. 可选择将受影响的设备或 DDS 子功能置于降级模式

## 5.4 安全模式下的运行

处于安全模式期间:未经明确的恢复动作,不会自动恢复至生产性执行动作;运维人员与支持团队以诊断为导向开展工作;每项操作都必须被记录;若存在物理原因的可能性,须先进行现场检查,再进行软件恢复。

## 5.5 从安全模式安全恢复

仅当满足以下全部条件时,方可恢复至正常运行:
- 根本原因已被消除,
- 配置/配置文件/会话已重新验证,
- 健康与状态均保持稳定,
- 已触发明确的恢复动作,
- 已通过一次简短的功能测试。

### 安全模式决策表

| 条件 | 处置动作 |
|----------------------------------------|-----------------------------------------------|
| 严重通信中断 | 激活安全模式 |
| 单条非严重命令出现故障 | 拒绝该命令,不进入安全模式 |
| 配置文件完整性被破坏 | 停止激活,若已处于激活状态则进入安全模式 |
| 仅在扩展路径上出现 DDS QoS 不匹配 | 隔离 DDS,南向运行继续 |
| 手动确认恢复成功 | 退出安全模式 |

该表汇总了源文档中所描述的故障与抑制原则,便于运维使用。

# 6. 运维最佳实践

## 6.1 监控

建议对所有状态域进行监控:生命周期、连接、协议、配置文件、设备类别、厂商专属,以及(若扩展功能处于激活状态)DDS 域。关键指标包括连接状态、故障率、恢复计数器、重启频率、发现稳定性、命令拒绝率以及审计事件。

## 6.2 日志与告警

日志应采用 JSON 结构化格式,可按严重程度过滤,并配备轮转/保留机制。至少需针对以下情况设置告警:严重故障、进入安全模式、回滚、签名错误、配置文件失效、DDS 发现丢失,以及反复出现的命令拒绝。

## 6.3 维护

维护工作宜在 `SERVICE_MODE`(维护模式)或 `READY`(就绪)状态下进行,而非在执行动作激活期间。每次维护活动之前,应备份当前配置、配置文件状态及故障历史记录。维护完成后,必须进行一次简化的功能测试及安全模式测试。

## 6.4 更新与回滚策略

每次更新都必须作为一次受控变更执行,包括预先验证、原子化激活、观察窗口期,以及明确定义的回滚阈值。固件与配置更新必须经过签名/验证;若出现错误,须立即恢复至上一个已知良好状态。

### 建议的变更流程

1. 批准变更
2. 验证工件
3. 开启维护窗口期
4. 原子化激活
5. 观察健康状况/故障/遥测数据
6. 执行冒烟测试
7. 关闭变更流程,或触发回滚

## 6.5 运维最低检查清单

### 日常运维
- 查看故障概览
- 检查安全模式事件
- 验证通信与发现机制的稳定性
- 查看审计与安全事件

### 每周例行工作
- 检查日志轮转与保留情况
- 评估恢复计数器与重启趋势
- 验证配置文件/固件兼容性
- 将 DDS 主题/QoS 目录与目标状态进行比对

### 生产发布前
- 完成配置验证
- 安全模式功能测试
- 回滚测试
- 重新加入/重连测试
- 完整的审计证据

# 7. 运行状态模型

```text
BOOTING(启动中)
  -> INITIALIZING(初始化中)
      -> READY(就绪)
          -> ACTIVE(运行中)
          -> SERVICE_MODE(维护模式)
          -> DEGRADED(降级)
          -> SAFE_MODE(安全模式) / FAULT_LOCKED(故障锁定)
```

该状态模型反映了 ICD 与 PSM 中所述的运行状态及抑制机制概念。

# 8. 本指南的局限性

源文档明确定义了架构、状态、治理机制及恢复原则;但它们**并未一贯包含具体的 REST 端点、CLI 命令或精确到位的协议帧格式**。因此,本指南特意采用了**与具体系统及接口无关的中立操作规程**表述方式,应在项目专属的运行手册中,补充实际的 API 名称、服务 ID、主题名称及设备模板。

# 9. 执行摘要

QNC 稳健的运维策略是:**严格分离已批准的工件、以原子化方式激活每一次变更、彻底隔离各状态域、在严重错误发生时立即切换至操作抑制/安全模式、以明确且可验证的方式执行恢复,并始终保持能够回滚至上一个已知良好状态的能力**。正是这一整套组合,共同创造了所需的部署安全性与运行时稳定性。
