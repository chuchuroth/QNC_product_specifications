## 二、核心知识地图（Knowledge Map）

**Git 知识体系结构**：

*   **版本控制基础**
    *   **Repository（仓库）**：受 Git 管理的文件夹，包含 `.git` 子文件夹。
    *   **Commit（提交）**：版本控制的基本单元，代表仓库某一时刻的快照。
        *   属性：Commit Message（摘要）、Commit ID/Hash（唯一标识）。
*   **本地管理流**
    *   **Init**：初始化仓库。
    *   **Gitignore**：定义排除规则，防止敏感或冗余文件入库。
    *   **Undo 操作**：Discard（放弃未提交改动） -> Reset（回退状态） -> Revert（生成反向提交，更安全）。
*   **分支与并发**
    *   **Branch（分支）**：不同的开发线（Master/Main 为主干）。
    *   **Merge（合并）**：将分支改动集成。
    *   **Worktree（工作树）**：支持多分支并行开发的物理副本。
    *   **Conflict（冲突）**：多方修改同一行时需人工或 AI 介入解决。
*   **远程与协作（GitHub）**
    *   **Remote Repository（远端仓库）**：本地仓库的备份与共享中心。
    *   **操作流**：Clone（下载） -> Pull（拉取并合并） -> Push（推送）。
    *   **多人协作模型**：
        *   **Fork**：克隆项目到个人名下。
        *   **Pull Request (PR)**：提交合并请求，触发 Code Review（代码审核）。

## 三、详细知识整理

### 1. 基础配置与初始化
*   **核心观点**：让 AI 辅助完成环境配置与项目初始化是高效起步的关键。
*   **为什么**：新手容易在 SSH 绑定、仓库关联等琐碎步骤上卡住，AI 可以自动执行这些底层命令。
*   **如何做**：
    1.  安装 Git 软件（Windows 官网下载，Mac 使用 `xcode-select --install`）。
    2.  注册 GitHub 账号并登录。
    3.  使用 VS Code 或 AI 编辑器（如 Cursor）打开空白文件夹。
    4.  **指令示例**：“把这个文件夹初始化成 git 项目，并推送到 GitHub 仓库 [仓库地址] 上”。
*   **注意事项**：`.gitignore` 文件应在初始化时创建，明确排除 `.env`（秘钥）和 `node_modules`（大型依赖）等目录。
*   **总结**：环境准备是第一步，通过 AI 绑定远端仓库可简化流程。

### 2. 提交（Commit）与“后悔药”机制
*   **核心观点**：Commit 是 Git 的核心快照，理解三种撤销操作的区别至关重要。
*   **为什么**：开发过程中不可避免会出错，Git 提供了不同层级的回退方案。
*   **操作与区别**：
    *   **Discard**：放弃**尚未 commit** 的本地改动。
    *   **Reset**：强制回回到某个历史 commit（有丢失代码风险，多人协作分支禁用）。
    *   **Revert**：生成一个**反向提交**来抵消某次改动（最安全，推荐协作时使用）。
*   **举例**：
    *   **背景**：发现“香蕉改菠萝”的提交是错的，但不想影响后续其他提交。
    *   **方法**：复制该提交的 Commit ID，告知 AI：“把这次提交撤销掉，保持其他的不变”。
    *   **结果**：AI 执行 `git revert`，生成反向操作，代码逻辑恢复。
*   **总结**：善用 Commit ID 和 AI，可以精准、安全地修正历史错误。

### 3. 分支管理与并行开发
*   **核心观点**：分支是实现多人/多任务并行而不互相干扰的机制。
*   **如何做**：
    1.  从 Master 创建 Feature 分支。
    2.  在 Feature 上开发并 Commit。
    3.  切回 Master，执行 Merge 将功能合并。
*   **进阶工具：Worktree**：
    *   **场景**：主干开发时需紧急处理另一任务，但不希望频繁切换 HEAD。
    *   **步骤**：创建 Worktree 副本文件夹 -> 在新文件夹并行开发 -> 合并回主干。
*   **冲突解决**：当两分支改动同一行，AI 会列出选项（保留 A、保留 B 或两者都保留），开发者根据业务逻辑选择即可。
*   **总结**：分支让开发更有条理，AI 让复杂的冲突解决变得简单。

### 4. 远端协作与 GitHub 深度使用
*   **核心观点**：GitHub 不仅仅是存储，更是基于 Fork 和 PR 的全球协作平台。
*   **关键术语/功能**：
    *   **Fork**：将他人的开源项目“复刻”一份到自己名下，获得修改权限。
    *   **Pull Request (PR)**：将你的 Fork 分支改动请求合并回原作者主干的提案。
    *   **Issues**：讨论 Bug、提出新功能建议的场所。
*   **GitHub 快捷键**：
    *   `/`：搜索。
    *   `t`：搜索文件。
    *   `l`：定位行号。
    *   `.`：在浏览器直接打开 Web 版 VS Code 查看代码。
*   **总结**：掌握 Fork 和 PR 流程是参与开源项目或团队开发的核心。

## 四、方法论整理

| 方法名称 | 适用场景 | 操作步骤 | 优点 | 注意事项 |
| :--- | :--- | :--- | :--- | :--- |
| **AI 辅助修复法** | 需要撤销已提交的特定错误 | 复制 Commit ID -> 自然语言指令给 AI（Revert/Reset） | 无需记命令，准确率高 | `Reset` 后需强推，协作分支慎用 |
| **Worktree 并行法** | 需要在不同分支间频繁切换或同时测试 | 创建独立工作树文件夹 -> 分别开发 -> 统一合并 | 物理隔离，不相互干扰，效率高 | 需及时删除已完成的工作树以节省空间 |
| **开源贡献 PR 流** | 为开源项目提交代码 | Fork -> 本地开发 -> 同步母项目改动 -> 解决冲突 -> 发起 PR | 标准化、专业化协作 | **发起 PR 前务必先同步母项目最新代码** |
| **Rebase 变基** | 追求简洁的提交历史 | 执行 Rebase 命令（更改地基） -> 合并分支 | 历史链条线性、干净，无冗余 Merge 记录 | 必选强推（Force Push），严禁在多人公共分支使用 |

## 五、概念词典

*   **Git**：开源免费的版本控制软件。
*   **GitHub**：全球最大的代码托管与协作平台（Git 仓库的中心/Hub）。
*   **Commit ID (Hash)**：通过哈希算法生成的唯一标识，用于精确指代某次提交。
*   **HEAD**：指向当前仓库所处的位置（通常是某个分支的最新提交）。
*   **Detached HEAD（分离头指针）**：当前状态不处于任何分支管理中，仅适合查看代码，不宜修改。
*   **Stash**：临时存储区，用于保存写了一半、尚未 Commit 的代码以便紧急切换任务。
*   **Cherry-Pick（拣选）**：从众多提交中“挑拣”特定的提交合并到当前分支。

## 七、知识关联

1.  **Repository** 包含多个 **Commit** 形成的链路。
2.  **Branch** 实际上是基于某个 **Commit** 创建的指针副本。
3.  **Conflict** 发生在 **Merge** 过程中，当两个分支修改同一位置时产生。
4.  **PR (Pull Request)** 依赖于 **Fork** 和 **Branch**，是协作的最终形式。
5.  **Rebase** 会导致分支根基变化，因此**依赖**于 **Force Push** 才能同步到远端。

## 九、一页知识卡（One-page Summary）

*   **一句话主题**：通过 AI 指挥 Git 进行高效版本管理。
*   **核心观点**：
    1.  Git 管理的是快照（Commit），GitHub 是协作中心。
    2.  AI 时代，理解“意图”比记忆“命令”更重要。
    3.  分支是并行开发的基础，Worktree 是进阶并行利器。
    4.  撤销操作中，Revert 比 Reset 更适合协作。
    5.  开源协作的核心是：Fork -> Branch -> Sync -> PR。
*   **最重要的方法**：使用自然语言 + Commit ID 指导 AI 进行回退、合并和冲突解决。
*   **最重要的框架**：Git 的四个分区模型（工作区、暂存区、本地仓库、远端仓库）。
*   **最重要的结论**：掌握 Git 核心逻辑后，AI 能帮你处理 90% 的底层操作，将你的精力释放到业务逻辑开发中。
