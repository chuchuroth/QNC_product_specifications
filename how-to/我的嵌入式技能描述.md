### 二、操作系统与系统原理

* 熟悉 **嵌入式系统的启动流程**：从上电复位 → BootLoader → Linux 内核 → 根文件系统 → 用户空间；
* 理解 **FreeRTOS 内核机制**：任务调度、任务切换、中断管理、内存分配、信号量、消息队列、互斥锁等；
* 能独立编写和调试 **多任务应用**，熟悉 FreeRTOS 任务栈、时间片轮转、临界区保护机制；
* 熟悉 **嵌入式 Linux 内核架构**，包括进程管理、内存管理、设备驱动模型、VFS 文件系统；
* 能够阅读和修改内核源码，理解 **内核初始化流程（start_kernel）** 与设备树（Device Tree）机制。

---

### 三、BootLoader（U-Boot）开发与移植

* 理解 BootLoader 的分阶段结构（SPL/MLO 与 main U-Boot）；
* 熟悉 **U-Boot 的启动流程**：硬件初始化 → DDR/串口初始化 → 加载内核 → 跳转执行；
* 掌握 **U-Boot 的编译、配置与裁剪**：

  * 使用 `make <board>_defconfig` 生成配置；
  * 通过 `menuconfig` 修改编译选项；
  * 删除不必要的命令和外设支持以减小镜像体积；
* 能独立完成 **U-Boot 的移植**：

  * 修改启动文件（board_init、lowlevel_init）；
  * 配置存储设备（NAND、eMMC、SD卡）与网卡驱动；
  * 修改环境变量存储方式（Flash、SD、NAND）；
* 熟悉 **Boot 参数传递机制**（bootargs）、镜像加载方式（TFTP、NFS、SD卡、Flash）；
* 能使用 U-Boot 命令行进行调试、镜像烧写与启动测试。

---

### 四、Linux 内核裁剪与移植

* 能通过 `make menuconfig` 进行 **内核功能裁剪**，剔除不必要模块（蓝牙、调试接口等）；
* 能根据目标硬件修改或新增 **设备树文件（.dts/.dtsi）**；
* 熟悉内核启动过程（从汇编入口 `_start` 到 `init` 进程）；
* 能修改启动参数、驱动加载顺序，调整内核映像格式（`zImage`、`uImage`）；
* 熟悉 **交叉编译内核** 及模块编译流程：

  ```bash
  make ARCH=arm CROSS_COMPILE=arm-linux-gnueabihf- zImage dtbs modules
  ```
* 理解 **内核模块（ko）** 加载机制，能编写简单字符设备驱动并注册设备节点；
* 能使用 `dmesg`、`insmod`、`rmmod`、`lsmod` 等命令进行驱动调试。

---

### 五、根文件系统（RootFS）与应用层开发

* 理解文件系统类型（ext4、jffs2、squashfs）及其在 Flash/SD 上的挂载方式；
* 能定制系统启动流程，实现自动启动脚本或后台守护进程。

### 六、硬件与底层控制

## 🧠 一、C语言的高级用法

C语言虽然语法简单，但可以用一些高级技巧写出灵活高效的代码：

| 高级用法            | 说明                           | 示例                                   |
| --------------- | ---------------------------- | ------------------------------------ |
| **指针**          | 操作内存地址，实现灵活访问                | `int *p = &x; *p = 10;`              |
| **结构体与typedef** | 自定义复杂数据类型                    | `typedef struct {...} Student;`      |
| **函数指针**        | 让函数也能像变量一样传递                 | `int (*f)(int,int); f = add;`        |
| **宏与预处理**       | 在编译前进行文本替换                   | `#define MAX(a,b) ((a)>(b)?(a):(b))` |

| **模块化编程**       | 用 `.h` + `.c` 文件分离接口与实现      | 提高可维护性                               |

## 🧩 二、C语言如何实现面向对象思想（OOP）

C语言本身不是面向对象的，但可以**用结构体 + 函数指针**模拟“对象”和“方法”概念。

举个例子：

```c
#include <stdio.h>

typedef struct {
    int x, y;
    void (*move)(int, int);
} Point;

void move_func(int dx, int dy) {
    printf("Move by (%d, %d)\n", dx, dy);
}

int main() {
    Point p = {10, 20, move_func};
    p.move(5, 3);  // 相当于“调用方法”
}
```

➡️ 这样我们就实现了：

* **数据（属性）**：`x, y`
* **函数（方法）**：`move`
* **对象（实例）**：`p`

**更高级一点：**

* “继承”可以通过结构体嵌套；
* “多态”可以通过函数指针数组实现（像C++的虚函数表）。

> 🔍 这就是为什么 Linux 内核、驱动、FreeRTOS 都用 C 实现，却有“面向对象”的感觉。

## ⚙️ 三、C语言如何控制硬件

C 之所以常用于嵌入式，是因为它能**直接操作内存地址**，从而控制硬件寄存器。

例如在 ARM 单片机中：

```c
#define GPIOA_ODR  (*(volatile unsigned int*)0x40020014)

int main() {
    GPIOA_ODR = 0x1;  // 直接写寄存器，点亮LED
}
```

解释：

* `0x40020014` 是硬件寄存器的地址；
* `volatile` 告诉编译器不要优化（寄存器值随时可能变化）；
* `*` 表示“取出该地址对应的内容”；
* 写入值就等于直接**控制硬件电路**。

所以：

> C语言控制硬件 = “访问特定内存地址的寄存器”。

## 🧮 四、C语言如何使用内存

内存分为几大区域：

| 区域                   | 存什么               | 由谁管理       |
| -------------------- | ----------------- | ---------- |
| **代码区（Text）**        | 程序指令              | 编译器        |
| **全局/静态区（Data/BSS）** | 全局变量、静态变量         | 系统         |
| **栈区（Stack）**        | 函数局部变量、参数         | 编译器自动分配/释放 |
| **堆区（Heap）**         | 动态内存（malloc/free） | 程序员自己管理    |

示例：

```c
int g = 10;            // 全局区
void func() {
    int a = 5;         // 栈区
    int *p = malloc(4);// 堆区
    *p = 20;
    free(p);
}
```

---

## 🧱 五、C语言代码在内存中的布局

编译后的可执行文件大致结构：

```
+---------------------+
| 代码区 (text)       | -> 函数、指令
+---------------------+
| 已初始化数据区      | -> int g = 10;
+---------------------+
| 未初始化数据区(BSS) | -> int b;
+---------------------+
| 堆 (heap)           | -> malloc分配
|                     |
| ...                 |
| 栈 (stack)          | -> 局部变量、函数调用
+---------------------+
```

内存从高地址往低地址增长：

* 栈从高地址往下长；
* 堆从低地址往上长；
* 两者中间的空间就是程序运行区。

## ⚡ 六、C语言代码在芯片中是如何执行的

整个过程如下 👇：

1. **编译（Compile）**
   `.c` 源代码 → 汇编代码 `.s` → 机器码 `.o`

2. **链接（Link）**
   把多个 `.o` 和库文件 `.lib/.a` 拼成可执行文件 `.elf`

3. **烧录（Flash）**
   将 `.elf` 转为二进制 `.bin` 写入芯片 Flash 存储区

4. **上电运行**
   CPU 从固定地址（例如 `0x00000000`）读取指令
   ↓
   程序计数器（PC）指向第一条指令
   ↓
   CPU 逐条取指 → 解码 → 执行
   ↓
   最终运行你的C函数（比如 `main()`）

## 🚀 二、系统启动流程（上电到应用启动）

   * 提供基本命令（ls、sh、cat等）；
   * 启动脚本运行应用程序；

* SD卡镜像写入（dd）
* 网络（TFTP）
* 专用烧录工具（如 J-Link、RKTool）

U-Boot中常用命令：

```bash
tftp 0x21000000 zImage
tftp 0x22000000 rootfs.jffs2
nand write 0x21000000 0x0 ${filesize}
```

---

## 📊 七、整体流程总结

| 阶段         | 主要任务          | 关键输出             |
| ---------- | ------------- | ---------------- |
| BootLoader | 初始化硬件，加载内核    | `u-boot.bin`     |
| 内核         | 控制硬件，挂载文件系统   | `zImage`, `.dtb` |
| 根文件系统      | 提供用户空间和命令环境   | `rootfs.ext4`    |
| 烧写启动       | 将三者写入Flash/SD | 系统可运行            |

---

✅ **一句话总结：**

> 嵌入式Linux系统的构建 = **BootLoader + 内核 + 根文件系统** 三者的编译、移植与集成，通过启动流程依次加载运行，最终形成一个可启动的嵌入式操作系统。

## 🧠 一句话总结

**FreeRTOS** 是一个**轻量级实时操作系统（RTOS）内核**，它的核心工作就是：

> 让多个“任务”在单个CPU上看起来像是同时运行，通过“任务切换（上下文切换）”来实现。

---

## 🔩 FreeRTOS的核心组成

FreeRTOS 主要由以下几个模块组成：

1. **任务管理（Task Management）**

   * 每个任务其实就是一个函数（带有自己的栈和状态）。
   * FreeRTOS 内部维护一个任务控制块（`TCB`，Task Control Block），保存任务的信息：

     * 栈指针
     * 优先级
     * 状态（运行、就绪、阻塞、挂起）
     * 延时计数等

2. **调度器（Scheduler）**

   * 负责决定“哪个任务现在该运行”。
   * 典型策略：**优先级抢占式调度（Preemptive Scheduling）**。
   * 调度算法简单高效：

     * 优先级高的任务先运行；
     * 如果有多个同优先级任务，轮流运行（时间片轮转）。

3. **时间管理（Tick Timer）**

   * 系统会设置一个定时中断，比如每 1ms 触发一次。
   * 每次 Tick：

     * 更新延时任务的计数；
     * 检查是否有任务该被唤醒；
     * 如果需要，则触发任务切换。

4. **任务切换（Context Switch）**

   * 切换任务其实是切换“栈指针”和“寄存器”。
   * 在中断（比如SysTick中断）中完成：

     * 保存当前任务寄存器 → 当前任务TCB；
     * 更新当前任务为新任务；
     * 从新任务TCB恢复寄存器 → 返回继续执行。

5. **队列、信号量、互斥量**

   * 提供任务间通信和同步机制。
   * 阻塞在这些对象上的任务会被移出“就绪列表”，直到资源可用。

6. **内存管理**

   * 提供几种内存分配策略（`heap_1` ~ `heap_5`），从最简单的静态分配到带碎片整理的动态分配。

---

## ⚙️ 一个运行过程的例子

假设你有三个任务：

* TaskA（高优先级）
* TaskB（中优先级）
* TaskC（低优先级）

1. 系统启动后，调度器选出最高优先级的任务（TaskA）执行；
2. TaskA 调用 `vTaskDelay(100)` 让自己睡 100 个tick；
3. 调度器切换到 TaskB；
4. 每次系统 Tick 中断，延时计数减1；
5. 当 TaskA 的延时结束时，它变为“就绪”；
6. 调度器立即抢占当前任务（TaskB），切回 TaskA；
7. 周而复始。

## 🧠 一句话解释

**FPGA（Field Programmable Gate Array）**：
是一种可以**由用户自己“编程”去实现硬件电路功能**的芯片。

也就是说——

> 用软件写“硬件逻辑”，让芯片按你的设计去变成各种不同的电路。

---

## ⚙️ 二、和普通芯片的区别

| 类型            | 特点             | 能否改功能？  |
| ------------- | -------------- | ------- |
| **CPU / MCU** | 已经固定功能（执行指令）   | ❌ 不行    |
| **ASIC**      | 为特定功能定制（如专用芯片） | ❌ 不行    |
| **FPGA**      | 内部结构可重新配置      | ✅ 可以反复改 |

💡 举例：

* MCU 是“写程序让芯片执行”；
* FPGA 是“写电路让芯片变形”。

---

## 🧩 三、FPGA 的内部结构

可以把 FPGA 想象成一个“电子乐高积木场”：

| 部分                 | 功能             |
| ------------------ | -------------- |
| **逻辑单元（LUT）**      | 实现基本逻辑（如与、或、非） |
| **触发器（Flip-Flop）** | 保存数据，实现时序逻辑    |
| **布线资源（Routing）**  | 把逻辑单元连起来       |
| **I/O 接口**         | 连接外部引脚或设备      |
| **时钟资源**           | 控制逻辑的运行节奏      |

通过“配置文件（bitstream）”，你可以告诉 FPGA：
这些逻辑单元怎么连、信号怎么流动。

---

## 🧰 四、FPGA 是怎么“编程”的

FPGA 不用 C 语言，而是用 **硬件描述语言（HDL）**：

* **Verilog HDL**
* **VHDL**

这些语言不是写“指令”，而是描述**电路逻辑**。
比如，你可以写：

```verilog
assign out = a & b;  // 逻辑与门
```

这不是让芯片“执行一次”，而是让芯片**内部生成一条与门电路**。

---

## 🔄 五、设计与运行流程（开发流程）

1. **编写 HDL 代码**（描述逻辑电路）
2. **综合（Synthesis）** → 转成逻辑门电路
3. **布局布线（Place & Route）** → 安排到具体硬件资源
4. **生成配置文件（Bitstream）**
5. **下载到 FPGA 芯片中运行**

💡 一旦烧进去，芯片就“变成”你设计的电路，
而且可以随时重新烧录，重新变形。

---

## ⚡ 六、FPGA 的常见应用

| 应用方向       | 示例             |
| ---------- | -------------- |
| **高速信号处理** | 视频编码、雷达信号、图像识别 |
| **通信系统**   | 5G基站、网络交换机     |
| **嵌入式加速**  | 与CPU配合，做并行运算   |
| **仿真与验证**  | ASIC 芯片验证      |
| **工业控制**   | 电机控制、传感器接口     |

---

| 项目   | CPU     | FPGA           |
| ---- | ------- | -------------- |
| 执行方式 | 按指令顺序执行 | 并行执行逻辑         |
| 灵活性  | 固定架构    | 可重构硬件          |
| 速度   | 通用较慢    | 并行运算快          |
| 功耗   | 相对低     | 通常更高           |
| 开发语言 | C / C++ | Verilog / VHDL |

## 🧱 一、ARM + Linux 开发是什么？

* **ARM**：一种广泛用于嵌入式设备（手机、路由器、工业控制板）的处理器架构。
* **Linux**：运行在 ARM 芯片上的操作系统。

> “ARM + Linux 开发” 就是：
> 在一块基于 ARM 架构的硬件上，使用 Linux 系统进行软件开发。

开发通常分为两部分：

1. **应用层开发（User Space）**
2. **驱动层开发（Kernel Space）**

## 🧩 二、应用层开发（User Space）

### 1️⃣ 所在层次

* 运行在 Linux 操作系统上；
* 和 PC 上开发 Linux 程序的方式类似；
* 用户可以直接看到和操作的部分（命令行程序、后台服务等）。

---

### 2️⃣ 开发内容

* 编写运行在用户空间的程序；
* 使用 **系统调用（system call）** 访问内核资源（文件、网络、设备等）；
* 可以使用 **C/C++** 或 **脚本语言（Python、Shell）**。

---

### 3️⃣ 典型任务

| 类型   | 示例                     |
| ---- | ---------------------- |
| 控制类  | 控制LED灯、读取传感器数据（通过设备文件） |
| 网络类  | 实现TCP/UDP通信            |
| UI类  | 嵌入式图形界面（Qt、GTK）        |
| 后台服务 | 开机自动运行的守护进程            |

---

### 4️⃣ 与硬件交互的方式

应用程序**不直接操作硬件**，而是通过：

* **文件系统接口**（如 `/dev/led0`）
* **系统调用**（如 `open()`、`read()`、`write()`）

💡 **例子：**

```c
int fd = open("/dev/led0", O_RDWR);
write(fd, "1", 1); // 点亮LED
close(fd);
```

> 你操作的是 `/dev/led0` 这个“文件”，
> 实际上这个文件背后对应的是**驱动程序控制的硬件**。

---

## ⚙️ 三、驱动层开发（Kernel Space）

### 1️⃣ 所在层次

* 运行在 **Linux 内核空间（Kernel Space）**；
* 直接和 **硬件寄存器** 打交道；
* 负责把“硬件细节”封装成“软件接口”。

---

### 2️⃣ 驱动的作用

驱动是操作系统和硬件之间的“翻译官”：

| 角色   | 职责                |
| ---- | ----------------- |
| 应用程序 | 想“点亮LED”          |
| 驱动程序 | 把“点亮LED”的请求转成硬件操作 |
| 硬件   | 真正发光的LED          |

---

### 3️⃣ 驱动开发常见类型

| 驱动类型   | 功能                          |
| ------ | --------------------------- |
| 字符设备驱动 | 通过 `/dev/xxx` 文件访问（如LED、串口） |
| 块设备驱动  | 存储设备（SD卡、eMMC）              |
| 网络设备驱动 | 网卡、Wi-Fi 模块                 |
| 总线驱动   | I²C、SPI、UART、USB 等接口驱动      |

---

### 4️⃣ 驱动开发常用操作

* 访问硬件寄存器（通过物理地址映射到内核虚拟地址）；
* 注册设备节点，让用户空间能通过 `/dev/...` 访问；
* 实现 `open() / read() / write() / ioctl()` 等回调函数；
* 使用内核提供的同步机制（中断、信号量、互斥锁）。

💡 **例子：最小字符设备驱动框架**

```c
static int led_open(struct inode *inode, struct file *file) {
    // 初始化硬件
    return 0;
}

static ssize_t led_write(struct file *file, const char *buf, size_t count, loff_t *ppos) {
    // 控制GPIO点亮/熄灭LED
    return count;
}

static struct file_operations led_fops = {
    .owner = THIS_MODULE,
    .open  = led_open,
    .write = led_write,
};

static int __init led_init(void) {
    register_chrdev(200, "led", &led_fops);
    return 0;
}
module_init(led_init);
```

编译后生成 `.ko` 文件，通过 `insmod led.ko` 加载到内核，就能在 `/dev/led` 下看到设备文件。

> 应用和驱动通过 `/dev` 文件和内核接口连接起来，
> 共同完成嵌入式系统的功能实现。
