# Contributing to Project Neura Robotics

Thank you for your interest in contributing to Project Neura Robotics!

## Getting Started

### Prerequisites

- **OS**: Linux (Ubuntu 20.04+ recommended) or development environment with Linux support
- **Compiler**: GCC 9+ with C++17 support
- **Build Tools**: CMake 3.16+, Make
- **Version Control**: Git

### Optional Dependencies
- **FastDDS 3.x**: For real DDS communication (otherwise uses mock)
- **libmodbus**: For Modbus hardware support (otherwise uses mock)
- **ROS 2**: For ROS integration (optional)

## Development Environment Setup

### 1. Clone the Repository
```bash
git clone <repository-url>
cd Project_Neura_Robotics
```

### 2. Install Dependencies
```bash
# Ubuntu/Debian
sudo apt update
sudo apt install -y build-essential cmake git
sudo apt install -y libfastdds-dev libfastcdr-dev  # Optional
sudo apt install -y libmodbus-dev                    # Optional
```

### 3. Build the Project
```bash
mkdir build && cd build
cmake .. -DCMAKE_BUILD_TYPE=Release
make -j$(nproc)
```

## Project Structure

```
Project_Neura_Robotics/
├── docs/           # All documentation (by topic)
```

## Coding Standards

### C++ Style Guide

1. **Naming Conventions**
   - Classes: `PascalCase` (e.g., `ModbusHandler`)
   - Functions: `camelCase` (e.g., `readRegister()`)
   - Variables: `snake_case` (e.g., `slave_address`)
   - Constants: `UPPER_SNAKE_CASE` (e.g., `MAX_RETRIES`)

2. **Code Formatting**
   - Indentation: 4 spaces (no tabs)
   - Line length: 100 characters max
   - Braces: Allman style
   ```cpp
   if (condition)
   {
       doSomething();
   }
   ```

3. **Comments**
   - Use `//` for single-line comments
   - Use `/** */` for documentation blocks
   - Explain "why", not "what"

4. **Error Handling**
   - Use exceptions for exceptional conditions
   - Return error codes for expected failures
   - Always log errors with context

## Git Workflow

### Branch Naming
- Feature: `feature/short-description`
- Bug fix: `bugfix/issue-description`
- Hot fix: `hotfix/critical-issue`
- Documentation: `docs/topic-name`

### Commit Messages
Follow the conventional commits format:
```
<type>(<scope>): <short description>

[optional body]

[optional footer]
```

Types:
- `feat`: New feature
- `fix`: Bug fix
- `docs`: Documentation changes
- `refactor`: Code refactoring
- `test`: Test additions/changes
- `chore`: Build/tooling changes

Examples:
```
feat(qnc): add IO-Link protocol support

Implements IO-Link master stack integration with device
descriptor parsing for automatic configuration.

Closes #123
```

```
fix(modbus): correct CRC calculation for write operations

The previous implementation used wrong polynomial for
multi-register writes.
```

### Pull Request Process

1. **Create Feature Branch**
   ```bash
   git checkout -b feature/your-feature-name
   ```

2. **Make Changes**
   - Write clean, well-documented code
   - Add/update tests as needed
   - Update documentation

3. **Test Locally**
   ```bash
   mkdir build && cd build
   cmake .. && make
   # Run tests
   ```

4. **Commit Changes**
   ```bash
   git add .
   git commit -m "feat(component): description"
   ```

5. **Push and Create PR**
   ```bash
   git push origin feature/your-feature-name
   ```

6. **PR Template** (Use [.github/PULL_REQUEST_TEMPLATE.md](.github/PULL_REQUEST_TEMPLATE.md))

## Documentation Guidelines

### Adding Documentation

1. **Choose the Right Location**
   - Architecture/design → `docs/architecture/`
   - Hardware specs → `docs/hardware/`
   - Protocols → `docs/protocols/`
   - Research → `docs/research/`
   - Operations → `docs/operations/`
   - Test reports → `docs/testing/`

2. **File Naming**
   - Use lowercase
   - Use underscores: `feature_name_description.md`
   - Be descriptive

3. **Markdown Format**
   - Use headers appropriately
   - Include table of contents for long documents
   - Add code blocks with language specification
   - Include diagrams where helpful

4. **Update Index**
   - Add links to main `docs/README.md`
   - Update relevant directory READMEs

## Testing

### Writing Tests

1. Tests should mirror source structure: `tests/component_name/`
2. Use descriptive test names
3. Test edge cases and error conditions
4. Mock external dependencies

### Running Tests
```bash
cd build
ctest --output-on-failure
```

## Code Review Guidelines

### As a Reviewer
- Be constructive and respectful
- Focus on code quality, not personal preference
- Check for security issues and edge cases
- Verify tests are adequate

### As an Author
- Respond to all comments
- Don't take feedback personally
- Explain your reasoning when disagreeing
- Update PR based on feedback

## Adding New Hardware Support

### For New Grippers/Devices

1. **Create Device Descriptor**
   ```json
   // config/devices/your_device.json
   {
     "device_type": "YourDevice_Model",
     "protocol": "modbus_rtu",
     "registers": {...},
     "commands": {...}
   }
   ```

2. **Document the Device**
   - Add to `docs/hardware/`
   - Include wiring diagrams
   - List tested configurations

3. **Add Tests**
   - Unit tests for protocol handling
   - Integration test with actual hardware
   - Document in `docs/testing/`

## Questions or Problems?

- Check existing documentation in `docs/`
- Review closed issues/PRs
- Ask in team chat
- Create a GitHub issue with:
  - Clear description
  - Steps to reproduce (for bugs)
  - Expected vs actual behavior
  - Environment details
