
## 产品定义

QNC（Quick Network Connector / Quick Neura Connector）被定义为一个软件定义的机器人网关与协议级多协议桥接器，位于机器人控制系统与工业末端执行器（如夹爪、传感器等）之间，作为统一的通信与集成中间层。

其核心目标是实现对工业现场多种异构通信协议（如 Modbus RTU、IO-Link 等）的抽象与统一管理，将底层物理层与传输层通信复杂性从机器人应用逻辑中解耦出来，从而显著降低设备接入成本并实现快速集成（可达“分钟级”接入能力）。

在架构上，QNC 负责协议转换、通信管理与接口规范化，并将不同设备协议统一映射为标准化的数据流（如 DDS 原生数据流）。与此同时，设备的专有语义与行为逻辑不再固化在驱动层或网关内部，而是通过外部的声明式 JSON 描述文件（描述符）进行定义与管理，由机器人端的执行逻辑或描述符驱动的执行引擎进行解析与处理。

因此，QNC 的本质是一个“协议桥接 + 数据流统一 + 声明式设备描述驱动”的基础设施层，它通过将设备通信、数据表达与语义建模解耦，减少传统设备驱动开发工作，使机器人系统能够以更高可配置性和可扩展性快速集成异构工业末端设备。

QNC（Quick Network Connector / Quick Neura Connector）在其体系中依赖两类核心规范文档——DPS 配置文档与 ICD 接口控制文档——共同定义设备抽象、通信行为与系统职责边界，从而构建统一的机器人-末端执行器集成框架。

在整体架构上，QNC 作为机器人控制系统与工业末端执行器之间的协议级软件定义网关，负责多协议通信（当前基于 Modbus RTU 与 IO-Link）、会话管理、数据标准化与配置文件驱动的行为执行，并将底层设备通信统一抽象为标准化的数据流与逻辑接口。机器人平台仅关注任务编排与运动决策，而设备则负责自身执行逻辑与物理行为，从而形成清晰的职责解耦。

在设备描述层面，DPS（Device Profile Specification）定义了外部设备的语义模型与能力抽象，其采用五层结构：身份层用于描述设备厂商、型号与类别；绑定层定义所使用的底层通信协议；能力层以设备功能为核心（如使能、定位、测量），屏蔽寄存器细节；映射层负责将标准化 QNC 指令与设备原生协议操作进行转换；治理层则管理版本、验证状态、签名与生命周期信息。

在执行模型上，DPS 明确规定指令必须具备原子性及可验证的成功/失败判定机制，支持单步或多步事务控制；遥测数据需统一转换为标准工程单位（推荐 SI 单位制），并附带质量标识（如有效、过时或估算）；所有设备故障必须显式映射至 QNC 的错误或告警体系，禁止静默忽略异常。

ICD（Interface Control Document）则进一步定义系统级接口契约与运行行为规范。在通信架构上，QNC 提供统一的逻辑通道，包括命令、响应、遥测、故障、生命周期与服务频道。它在指令执行前进行结构校验、参数验证与模式一致性检查，并在不满足条件时直接拒绝执行。

系统状态方面，QNC 维护独立的生命周期与运行状态模型，包括 BOOTING、INITIALIZING、READY、ACTIVE、DEGRADED、SAFE_MODE 与 FAULT_LOCKED 等阶段，用于精确描述系统健康度与可用性。在故障管理上，系统将异常划分为配置、协议、连接等类别，并按 INFO 至 CRITICAL 进行分级处理；同时明确 QNC 不属于安全认证系统（non-safety-rated），仅负责故障报告与安全模式切换，而不具备 E-stop 决策权。

在配置与治理方面，每个描述文件必须包含完整元数据（ID、版本、设备类别、协议绑定、初始化流程、能力集与映射关系），并在加载与激活前通过签名验证与完整性检查，否则将被拒绝执行。同时，系统要求完整覆盖指令一致性、遥测一致性、故障处理、安全模式切换与描述文件行为验证等测试范围，以保证系统级一致性与可靠性。

综合来看，DPS 定义“设备如何被抽象与描述”，ICD 定义“系统如何通信与运行”，二者共同支撑 QNC 实现一个以声明式配置驱动、强校验、强约束、协议抽象化的机器人工业设备接入中间层。

# 软件架构设计的考虑点

直接做“一份能装下所有夹爪的通用 IDL”在工程上不太现实，更推荐做“抽象接口 + 扩展字段”的分层设计。 

## 为什么“一个大而全 IDL”问题很大

不同夹爪差异非常大：  
- 驱动方式：两指、电动三指、气动、软体、真空吸盘等。 
- 控制模式：位置控制、力控制、速度控制、开/关、预设抓取模式等。 
- 反馈信息：是否夹紧成功、当前位姿、当前力、指尖触觉阵列、温度等。 

如果硬凑成一个 IDL：  
- 会有一大堆字段只对某些夹爪有意义，对其他夹爪永远是空。  
- 一旦遇到新型夹爪（比如多指灵巧手、软体手），又要改这个“通用 IDL”，所有下游代码都要重新生成、重新编译，兼容性很差。 

这类问题在机械手、夹爪研究里也常见，文献和工业实践更偏向“模块化接口/标准接口”，而不是“一包打天下的数据结构”。 

## 更推荐的做法：抽象控制接口 + 能力发现

可以把思想拆成三层：协议思想是统一的，具体数据结构可以用若干个 IDL 类型来实现。

### 1）抽象的通用控制命令/状态

定义一个尽量“瘦”的、抽象层的 IDL，比如只管这些通用概念：  
- 通用命令：  
  - 激活/复位  
  - 打开到某个开度（0~1 归一化）  
  - 施加某个力/力矩上限（如果支持）  
  - 停止/急停  
- 通用状态：  
  - 当前开度（0~1）  
  - 当前抓取状态：空闲、运动中、抓取成功、滑落、错误  
  - 当前错误码  

类似 ROS 里很多夹爪驱动最后都提供“position + max_effort”这样统一的 action/service 接口，不同厂商驱动在内部做适配。 

可以用一个 IDL 定义这种“逻辑通用接口消息”，比如：  
- `GripperCommand`：目标开度、目标力上限、模式枚举  
- `GripperState`：当前开度、力估计、状态枚举、错误码  

### 2）能力描述（Capabilities）

不同夹爪支持的功能不一样，需要一种“能力发现/描述”机制：  
- 夹爪是否支持力控制、速度控制、多指独立控制等。 
- 有多少个自由度、每个自由度的行程范围。  
- 是否有触觉、距离传感器等。  

可以单独做一个 `GripperCapabilities` 的 IDL：  
- 布尔或枚举字段：`has_force_control`、`has_multi_finger` 等。  
- 数组/列表：每个关节的最小/最大位姿、最大力。  

上层控制逻辑先订阅能力消息，按能力决定用哪一部分通用命令；如果某个功能不支持，就禁用对应 UI/算法分支。

### 3）厂商/设备特定扩展

对于某些高阶夹爪（软体、多指灵巧手、带密集触觉阵列），通用接口肯定不够用，这时：  
- 允许厂商自定义自己的 IDL 类型，比如 `VendorXGripperCommand`、`VendorXGripperState`，只在需要时用。  
- 在系统里，可以同时支持：  
  - 通用接口（方便快速接各种简单夹爪）；  
  - 专有接口（需要高级功能时，用对应驱动和消息）。

很多 ROS 驱动就是这么做的：对外暴露一个简单的统一 service/action，同时还提供一些更底层的话题或 service 暴露原始寄存器/高级参数。 


## 实现建议（从“想法”到“工程落地”）

建议这样设计：

1. 先列出想支持的“最小通用功能集”：  
   - 最常见的是“开度控制 + 可选力限制 + 状态反馈”，参考 PR2 gripper、Robotiq gripper 等接口设计。 

2. 基于这个功能集写第一版 IDL：  
   - `GripperCommand`  
   - `GripperState`  
   - `GripperCapabilities`  
   保持字段少而明确，先不要把所有想象得到的字段都塞进去。  

3. 对每一款具体夹爪，写一个“适配层”：  
   - 下游逻辑永远只面对通用 IDL 类型；  
   - 适配层负责把通用命令翻译成：Modbus/CAN/串口/TCP/IP/厂商 SDK 调用，并从底层反馈转成 `GripperState`。类似很多论文里说的“接口层/通信层”设计。 

4. 如果遇到某个夹爪确实有通用接口抽象不了的东西：  
   - 要么扩展 `GripperCapabilities` 和通用消息（小心演进）；  
   - 要么增加一个“厂商特定” IDL，专门给这类设备用。

这样，“通用”的其实是抽象接口，而不是一个“把所有细节全塞进去的超级数据结构”。

## 核心问题

- 出发点是对的：做通用抽象、屏蔽不同夹爪差异，这是非常标准的机器人系统工程思路。 
- 但实现形式不建议是“一个巨大无比的万能 IDL”，而是“少量抽象 IDL 类型 + 能力描述 + 厂商扩展 + 适配层”。

# 多协议通讯网关架构设计思路

以下是针对**多协议机器人通讯网关**的三种架构模式的完整设计思路。每种模式都从**数据流、内部模型、任务划分、关键实现、适用场景**五个维度展开。

---

# 方案一：协议转换器 (Protocol Converter)

## 核心理念
> **A口进，B口出**。网关作为两个（或多个）异构网络之间的翻译官，每个方向有明确的转换规则。

## 1. 数据流架构

```
┌─────────┐      ┌─────────────┐      ┌─────────┐
│ CANopen │◄────│  协议转换器  │────►│PROFINET│
│  网络   │      │   (网关)    │      │  网络   │
└─────────┘      └──────┬──────┘      └─────────┘
                        │
                   ┌────▼────┐
                   │ 可选的  │
                   │ 第三网络│
                   └─────────┘
```

**典型流向**：
- **正向转换**：CANopen PDO → 内部格式 → PROFINET IO 数据
- **反向转换**：PROFINET 控制字 → 内部格式 → CANopen SDO/PDO
- **旁路转发**：某些信号直接透传（如急停）

## 2. 内部数据模型（轻量级）

**不维护完整过程映像**，只维护**转换映射表**：

```c
typedef struct {
    uint32_t source_protocol;   // 0=CANopen, 1=PROFINET, 2=Modbus
    uint32_t source_addr;       // CANopen: 索引+子索引; PROFINET: slot+subslot
    uint32_t target_protocol;
    uint32_t target_addr;
    uint8_t  direction;          // 0=单向, 1=双向
    uint8_t  scaling_type;       // 线性/查表/自定义函数
    float    scale_factor;
    float    offset;
    uint32_t deadband;           // 变化超过此值才转发
    uint32_t last_value;
    uint32_t last_timestamp;
} ConversionRule;
```

**特点**：
- 没有统一的对象字典，只有一张转换规则表（通常几十到几百条）
- 规则在启动时从配置文件加载，运行时只增删改（极少）
- 每个规则独立处理，互不干扰

## 3. 任务划分

| 任务名 | 优先级 | 周期 | 职责 |
| :--- | :--- | :--- | :--- |
| **CANopen 接收** | 高 (3) | 1ms | 接收PDO，放入原始数据队列 |
| **PROFINET 接收** | 高 (3) | 1ms | 接收IO数据，放入原始数据队列 |
| **转换引擎** | 中 (2) | 2ms | 轮询队列，查表执行转换，产生输出指令 |
| **CANopen 发送** | 中 (2) | 异步 | 发送SDO/PDO到CAN网络 |
| **PROFINET 发送** | 中 (2) | 周期性 | 更新IO输出缓冲区 |
| **配置/诊断** | 低 (0) | 100ms | 处理规则配置、输出诊断信息 |

## 4. 关键实现细节

### 4.1 转换规则存储
```
配置文件 (JSON/XML/二进制)
    ↓ 启动时加载
内存中的转换规则数组 (静态分配，最大N条)
    ↓ 运行时CRC校验
快速查找哈希表 (key = source_protocol + source_addr)
```

### 4.2 数据一致性保证
```c
// 使用双缓冲 + 版本号
struct {
    ConversionRule rules[MAX_RULES];
    uint32_t version;      // 每次修改递增
    uint32_t crc32;
} rule_set_active, rule_set_shadow;

// 修改规则时
memcpy(&rule_set_shadow, &rule_set_active, sizeof);
// 修改 rule_set_shadow.rules[...]
rule_set_shadow.version++;
rule_set_shadow.crc32 = calc_crc(&rule_set_shadow);
// 原子切换指针
active_rules = &rule_set_shadow;
```

### 4.3 死区/滤波机制
```c
bool should_forward(uint32_t new_value, ConversionRule* rule) {
    uint32_t delta = abs((int32_t)new_value - (int32_t)rule->last_value);
    if (delta > rule->deadband) {
        rule->last_value = new_value;
        return true;
    }
    return false;  // 变化太小，不转发
}
```

## 5. 适用场景

| 场景 | 是否推荐 | 理由 |
| :--- | :--- | :--- |
| 连接两个明确的异构网络（如CANopen伺服 → PROFINET PLC） | ✅ 最佳 | 转换规则清晰，数据量可控 |
| 协议网关产品（如Anybus、EtherCAT-Junction） | ✅ 典型 | 市场成熟，有现成方案参考 |
| 机器人控制器内部多个子系统互连 | ⚠️ 一般 | 规则数可能膨胀到上千条，管理复杂 |
| 需要任意设备间相互通信 | ❌ 不适合 | 缺乏灵活性，每增加一对设备需新增规则 |

# 方案二：数据总线 (Data Bus)

## 核心理念
> **发布-订阅**。所有协议栈平等地读写一个共享的“过程映像”，网关只是总线的载体。

## 1. 数据流架构

```
                    ┌─────────────────────────────────┐
                    │        过程映像 (Process Image) │
                    │  ┌─────┐ ┌─────┐ ┌─────┐        │
                    │  │点1  │ │点2  │ │点3  │ ...    │
                    │  └──▲──┘ └──▲──┘ └──▲──┘        │
                    │     │       │       │           │
                    └─────┼───────┼───────┼───────────┘
                          │       │       │
    ┌─────────┐     ┌─────┴───────┴───────┴─────┐     ┌─────────┐
    │ CANopen │     │       数据总线核心        │     │PROFINET│
    │  设备   │◄───►│  • 读写仲裁              │◄───►│   PLC   │
    └─────────┘     │  • QoS管理               │     └─────────┘
    ┌─────────┐     │  • 时间戳/质量戳         │     ┌─────────┐
    │ Modbus  │◄───►│  • 持久化(可选)          │◄───►│EtherNet/│
    │  设备   │     └──────────────────────────┘     │   IP    │
    └─────────┘                                      └─────────┘
```

## 2. 内部数据模型（完整过程映像）

```c
// 数据点描述符
typedef struct {
    uint32_t id;                    // 唯一标识符 (hash from name)
    char     name[64];              // "robot.joint1.position"
    uint8_t  type;                  // INT8/UINT16/FLOAT32/STRING...
    uint8_t  access;                // RO/RW/RW_EXCLUSIVE
    uint32_t size;                  // 字节数
    void*    storage;               // 指向实际存储位置
    uint32_t timestamp_ns;          // 最后更新时间 (单调时钟)
    uint8_t  quality;               // 0=BAD, 1=UNCERTAIN, 2=GOOD
    uint8_t  protocol_source;       // 哪个协议写的
    uint32_t subscriber_mask;       // 哪些协议订阅了这个点
} DataPoint;

// 过程映像管理
typedef struct {
    DataPoint*   points;            // 数据点数组
    uint32_t     point_count;
    SemaphoreHandle_t rw_lock;      // 读写锁（支持多读单写）
    uint32_t     global_version;    // 整个映像版本号
} ProcessImage;
```

## 3. 任务划分

| 任务名 | 优先级 | 周期 | 职责 |
| :--- | :--- | :--- | :--- |
| **CAN发布者** | 高 (3) | 1ms | 读CAN PDO → 写过程映像 |
| **CAN订阅者** | 中 (2) | 2ms | 读过程映像 → 发送CAN PDO |
| **PROFINET发布者** | 高 (3) | 1ms | 读PROFINET IO → 写过程映像 |
| **PROFINET订阅者** | 中 (2) | 2ms | 读过程映像 → 更新PROFINET输出 |
| **QoS监视器** | 中 (1) | 10ms | 检查每个点的更新频率，标记STALE |
| **持久化任务** | 低 (0) | 1000ms | 将关键点保存到Flash（可选） |
| **诊断任务** | 低 (0) | 100ms | 发布诊断信息点 |

**关键设计**：发布者和订阅者是**分离的任务**，这样允许不同协议有不同的读写周期。



# 方案三：混合模式 (Hybrid)

## 核心理念
> **关键路径走总线，非关键/边界场景走转换**。根据数据类型动态选择最优路径。

## 1. 数据流架构

```
                              ┌─────────────────────────────────────┐
                              │           路由决策引擎              │
                              │  ┌──────────┐  ┌──────────────┐    │
                              │  │规则引擎  │  │动态优先级   │    │
                              │  │(静态配置)│  │(运行时学习) │    │
                              │  └────┬─────┘  └──────┬───────┘    │
                              └───────┼────────────────┼───────────┘
                                      │                │
                    ┌─────────────────┼────────────────┼─────────────────┐
                    │                 │                │                 │
                    ▼                 ▼                ▼                 ▼
            ┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────┐
            │ 实时总线    │    │ 高速数据流  │    │ 配置/诊断   │    │ 异常    │
            │ (过程映像)  │    │ (零拷贝通道)│    │ (转换器)    │    │ 旁路    │
            └──────┬──────┘    └──────┬──────┘    └──────┬──────┘    └────┬────┘
                   │                  │                  │                │
                   └──────────────────┼──────────────────┼────────────────┘
                                      │                  │
                              ┌───────▼──────────────────▼───────┐
                              │         协议栈适配层             │
                              │  CAN │ PROFINET │ Modbus │ ETH/IP │
                              └──────────────────────────────────┘
```

**三通道并行**：
- **通道1（实时总线）**：周期性控制数据（位置、速度、扭矩）→ 走过程映像 + QoS监控
- **通道2（高速直通）**：海量数据（点云、图像）→ 走零拷贝共享内存，不经过过程映像
- **通道3（配置/诊断）**：非周期性参数 → 走协议转换器模式，延迟不敏感

## 4. 关键实现细节

### 4.1 路由决策规则引擎
```c
// 静态规则表 (编译时可配置)
typedef struct {
    uint32_t signal_id;          // 如 "joint1_position"
    uint8_t  data_size;          // 字节数
    uint8_t  max_latency_us;     // 最大允许延迟
    uint8_t  min_period_us;      // 最小周期
    uint8_t  route_type;         // 0=实时映像, 1=高速流, 2=转换器
    uint8_t  fallback_route;     // 超限时的备选路径
} RoutingRule;

// 动态优先级调整 (运行时学习)
void routing_decision_maker(void* arg) {
    while(1) {
        // 监控每个通道的当前负载和延迟
        uint32_t bus_load = get_bus_utilization();
        uint32_t avg_latency = get_avg_latency_us();
        
        // 动态调整路由
        for each signal {
            if (signal.current_latency > signal.max_allowed_latency) {
                // 升级到更快通道
                if (signal.route != FASTEST_ROUTE) {
                    promote_signal_to_faster_channel(signal);
                }
            } else if (signal.current_latency < signal.min_latency && 
                       bus_load > 80) {
                // 延迟充裕，降级到较慢通道以释放资源
                demote_signal_to_slower_channel(signal);
            }
        }
        
        vTaskDelay(pdMS_TO_TICKS(10));
    }
}
```

### 4.2 高速流零拷贝实现
```c
// 生产者 (如网卡驱动接收到点云数据)
uint32_t produce_high_speed_frame(uint8_t stream_id, void* data, uint32_t len) {
    HighSpeedStream* stream = &streams[stream_id];
    
    // 检查是否有足够空间
    uint32_t available = (stream->read_index - stream->write_index - 1) % stream->buffer_size;
    if (available < len) {
        return ERR_BUFFER_FULL;
    }
    
    // 直接写入环形缓冲区 (无锁，使用原子操作)
    uint32_t pos = stream->write_index;
    memcpy(&stream->buffer[pos], data, len);
    
    // 发布写入完成 (原子更新)
    atomic_store(&stream->write_index, (pos + len) % stream->buffer_size);
    
    // 通知消费者 (可选，如果消费者在等待)
    xSemaphoreGive(stream->data_ready_sem);
    
    return len;
}

// 消费者 (如另一个协议栈要转发此数据)
void* consume_high_speed_frame(uint8_t stream_id, uint32_t* len) {
    HighSpeedStream* stream = &streams[stream_id];
    
    // 等待新数据 (带超时)
    if (xSemaphoreTake(stream->data_ready_sem, pdMS_TO_TICKS(10)) == pdTRUE) {
        uint32_t read_pos = atomic_load(&stream->read_index);
        uint32_t write_pos = atomic_load(&stream->write_index);
        
        if (read_pos != write_pos) {
            *len = (write_pos - read_pos) % stream->buffer_size;
            void* data = &stream->buffer[read_pos];
            
            // 注意：不拷贝，直接返回指针！调用方必须及时处理
            // 并在使用完后 atomic_update(&stream->read_index, ...)
            return data;
        }
    }
    return NULL;
}
```

### 4.3 异常旁路（当首选路径失败时）
```c
// 监控实时映像的健康状态
void health_monitor_task(void* arg) {
    uint32_t last_good_version = 0;
    uint32_t stale_counter = 0;
    
    while(1) {
        uint32_t current_version = realtime_image.version;
        if (current_version == last_good_version) {
            stale_counter++;
            if (stale_counter > 10) {  // 10ms无更新
                // 触发旁路：将关键信号切换到转换器模式
                activate_fallback_for_critical_signals();
            }
        } else {
            stale_counter = 0;
            last_good_version = current_version;
        }
        
        vTaskDelay(pdMS_TO_TICKS(1));
    }
}

// 旁路动作：将控制指令通过SDO/Modbus直接写入设备
void activate_fallback_for_critical_signals() {
    // 告诉路由决策引擎，临时走通道3（转换器模式）
    routing_table[CTRLWORD_SIGNAL].route_type = 2;  // 强制转换器
    routing_table[TARGET_POS_SIGNAL].route_type = 2;
    
    // 启动看门狗，5秒后尝试恢复
    start_recovery_timer(5000);
}
```

# 三种方案对比总结

| 维度 | 协议转换器 | 数据总线 | 混合模式 |
| :--- | :--- | :--- | :--- |
| **核心数据流** | 点对点映射 | 发布-订阅 | 动态路由 |
| **数据模型复杂度** | 低（映射表） | 高（完整映像） | 极高（分层+规则） |
| **代码规模** | 5k-15k行 | 15k-30k行 | 30k-60k行 |
| **RAM需求** | 256KB-512KB | 1MB-4MB | 2MB-8MB |
| **实时性（最小延迟）** | 100-500μs | 200-1000μs | 50-500μs（可优化） |
| **确定性** | 高（转换延迟固定） | 中（受锁竞争影响） | 高（关键路径固定） |
| **扩展性** | 差（每增加设备要加规则） | 好（只需写适配器） | 极好（自动路由） |
| **调试难度** | 低 | 中 | 高 |
| **开发周期** | 2-4个月 | 4-8个月 | 8-12个月 |
| **成熟度** | 极成熟（无数网关产品） | 成熟（类似ROS） | 前沿（自定义） |

```
阶段1 (1-2月)
    └── 实现协议转换器的最小原型
        ├── CANopen ↔ 内部表格
        ├── Modbus ↔ 内部表格
        └── 验证基础转发正确性
                │
                ▼

    └── 升级为数据总线模式

        ├── 实现QoS和质量戳
        ├── 添加第三个协议
        └── 压力测试 (1000点, 1kHz)
                │
                ▼

    └── 可选：增加混合模式特性
        ├── 高速流通道 (用于视觉/点云)
        ├── 动态路由规则
        ├── 异常旁路
        └── 性能调优
```

**如果只能选一个方案起步，我建议从方案二（数据总线）开始**，原因是：
- 协议转换器后期扩展规则爆炸，重构成本高
- 数据总线的架构一旦搭好，增加新协议只是写适配器
- 混合模式可以基于数据总线逐步增加，不需要推翻重来

根据项目具体情况，比如大约需要连接多少种协议、多少设备、数据量多大、实时性要求多高，根据这些参数来判断哪个方案最适合qnc。

这正是Neura Robotics所说的"机器人可以**看见、听见、感知**"的体现。

你提到SmartWrist必须"适配市面上大多数EOAT"，这个设计原则非常关键。它意味着：

这正是"设备即平台"的精髓——**平台不是通过排他性锁定客户，而是通过兼容性吸引客户，然后通过数据锁定客户**。
