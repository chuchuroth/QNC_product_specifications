# Protocol Detection Methods - Technical Analysis

This document analyzes the primary methods for detecting which device is connected to QNC and which protocol it uses.

## Overview

Three primary methods are considered:

1. Hardware ID pins (resistor-based identification)
2. Protocol probing (active communication testing)
3. Descriptor-based detection (configuration file matching)

## Method 1: Hardware ID Pins

### Concept

Each gripper has a unique resistor value on a dedicated ID pin. QNC reads the ADC value and maps it to a device/profile.

### Electrical Model

- Pull-up resistor on QNC board (example: 10kΩ)
- Device-specific ID resistor on tool side
- ADC reads divider voltage:

$V_{ADC} = V_{CC} \times \frac{R_{ID}}{R_{PULLUP} + R_{ID}}$

### Example Mapping

| Device | R_ID | ADC (12-bit) | Voltage |
|---|---:|---:|---:|
| DH-Robotics AG95 | 2.2kΩ | 1862 | 1.49V |
| Robotiq 2F-85 | 3.3kΩ | 2275 | 1.83V |
| Zimmer GEP5000IL | 4.7kΩ | 2559 | 2.05V |
| OnRobot RG2 | 6.8kΩ | 2856 | 2.29V |
| Schunk EGP64 | 10kΩ | 3072 | 2.46V |
| Unknown/Custom | Open | 4095 | 3.30V |

### Pros

- Instant detection (no bus traffic)
- Reliable and deterministic
- Works before full protocol stack initialization
- Very low BOM cost

### Cons

- Requires connector/hardware design control
- Limited scalability via ADC tolerance windows
- Hard to retrofit existing third-party devices
- Identifies profile/model, not protocol behavior details

### Recommendation

**Rating:** ★★★☆☆ (3/5)

Use when connector hardware is under your control and the device set is fixed.

## Method 2: Protocol Probing

### Concept

QNC actively probes supported protocols and classifies the connected device by valid responses.

### Typical Probe Flow

1. Try Modbus RTU
2. Try IO-Link
3. Try EtherCAT / other supported protocols
4. Select first successful result (or best-scored match)

### Modbus RTU Example Strategy

- Baudrates: `115200`, `19200`, `9600`
- Slave IDs: `1`, `2`, `3`
- Registers: common probes such as `0`, `512`, `1000`
- Validate response shape and CRC

### IO-Link Example Strategy

- Configure required GPIO/UART
- Send wake-up sequence
- Attempt master command exchange
- Validate response timing/format

### Pseudocode (Illustrative)

```cpp
ProtocolType probe_all_protocols() {
    if (probe_modbus_rtu("/dev/ttyAMA0", 115200, 100)) return MODBUS_RTU;
    if (probe_modbus_rtu("/dev/ttyAMA0", 19200, 100)) return MODBUS_RTU;
    if (probe_iolink("/dev/ttyAMA0", 200)) return IOLINK;
    if (probe_ethercat("eth0", 500)) return ETHERCAT;
    return UNKNOWN;
}
```

### Timing

- Single protocol probe: ~50–200ms
- Multi-protocol pass: ~200–1000ms
- With retries/fallbacks: up to ~3s

### Pros

- Works with standard connectors
- Supports unknown/third-party devices
- Can infer operational parameters (baudrate, slave ID)
- No manual descriptor required for baseline connection

### Cons

- Slower and non-deterministic vs static methods
- Requires powered and responsive device
- Probe traffic may conflict with some devices

### Optimization Techniques

- Probe last-known protocol first
- Early terminate on valid match
- Probe independent interfaces in parallel
- Cache last successful profile

### Recommendation

**Rating:** ★★★★★ (5/5)

This is the best general-purpose default for QNC firmware.

## Method 3: Descriptor-Based Detection

### Concept

Use a descriptor JSON to provide protocol and connection settings up-front, bypassing auto-detection.

### Descriptor Example

```json
{
  "device": {
    "manufacturer": "DH-Robotics",
    "model": "AG-95",
    "protocol": "modbus_rtu"
  },
  "connection": {
    "interface": "rs485",
    "port": "/dev/ttyAMA0",
    "baudrate": 115200,
    "slave_id": 1,
    "timeout_ms": 500
  }
}
```

### Load Process

1. Resolve descriptor path (CLI/config/env/default)
2. Parse JSON
3. Apply protocol/connection parameters
4. Initialize and connect

### Pros

- Fast and deterministic
- Rich device metadata available
- No detection ambiguity
- Easy expansion by adding new JSON descriptors

### Cons

- Depends on accurate, maintained descriptors
- Less automatic unless paired with probing
- Manual setup overhead in dynamic/hot-swap environments

### Recommendation

**Rating:** ★★★★☆ (4/5)

Excellent for production systems with known fleets and strict reproducibility.

## Recommended QNC Strategy (Hybrid)

1. Descriptor check (`<10ms`)
2. Hardware ID check (`<1ms`, if available)
3. Protocol probing (`200–1000ms`)
4. Cache result and monitor for hot-swap changes

### Example Config Sketch

```yaml
detection:
  enable_hardware_id: true
  enable_protocol_probe: true
  enable_descriptor: true
  default_descriptor: "/etc/qnc/devices/default.json"
  probe_timeout_ms: 500
  probe_order: ["modbus_rtu", "iolink", "ethercat"]
  cache_detection: true
  hot_swap_monitoring: true
```

This hybrid approach provides strong compatibility, predictable startup behavior when descriptors exist, and robust fallback discovery when they do not.
