

## Runtime Validation Routine for Gripper Hardware Communication
 
### This document describes the gripper runtime validation framework that confirms hardware responsiveness and validates command/state correlation at runtime on the QNC bridge.


# Gripper Runtime Validation Framework

## Overview

The runtime validation framework confirms that gripper hardware is responsive
and behaving correctly by:

1. **Publishing test commands** to the `qnc/gripper/command` DDS topic
2. **Monitoring state feedback** on the `qnc/gripper/state` DDS topic
3. **Validating parameters** match between command and state (within tolerance)
4. **Handling timeouts** gracefully if hardware doesn't respond
5. **Logging all interactions** for debugging and correlation

The framework is designed for:
- Single-device testing (default: DH56)
- Extensibility to multiple grippers
- Integration into CI/automated testing pipelines
- On-robot diagnostics and troubleshooting

## Architecture

### Components

**Header:** `include/gripper_runtime_validator.hpp`
- `GripperRuntimeValidator` class: Main validation orchestrator
- `ValidationResult` struct: Result details, diagnostics, timing
- `ValidationStatus` enum: Status codes (SUCCESS, TIMEOUT, MISMATCH, etc.)
- Helper function: `validate_gripper_device()` for quick single-device testing

**Implementation:** `src/gripper_runtime_validator.cpp`
- DDS publisher/subscriber setup via NeuraSync
- Timeout-aware state reception with condition variables
- Parameter tolerance matching (position, force, speed)
- Comprehensive logging with timestamps

**Executable:** `src/gripper_validation_test.cpp`
- CLI tool for running validation tests manually
- Supports single commands and multi-command sequences
- Configurable timeouts and tolerances
- Supports custom command parameters

### Design for Multi-Gripper Extension

The validator is designed to support multiple grippers by:

1. **Device ID parameter**: Each validator instance is bound to a single device
   ```cpp
   GripperRuntimeValidator validator_dh56("DH56");
   GripperRuntimeValidator validator_cgc80("CGC-80");
   ```

2. **Topicization by device**: Commands go to `qnc/gripper/command` (keyed by device_id),
   state comes from `qnc/gripper/state` (keyed by device_id). The DDS key-based
   topic design naturally handles multiple devices.

3. **Independent validation threads**: Each device can be validated in parallel:
   ```cpp
   std::vector<std::thread> validators;
   validators.push_back(std::thread([&]() {
       auto r1 = validator_dh56.validate_responsiveness();
   }));
   validators.push_back(std::thread([&]() {
       auto r2 = validator_cgc80.validate_responsiveness();
   }));
   for (auto& t : validators) t.join();
   ```

## Programmatic Usage (C++ API)

### Simple Validation

```cpp
#include "gripper_runtime_validator.hpp"
using namespace qnc::gripper;

// Quick check
auto result = validate_gripper_device("DH56", 2000, true);
if (result.is_valid()) {
    std::cout << "✓ DH56 is responsive\n";
} else {
    std::cerr << "✗ DH56 validation failed: " << result.message << "\n";
}
```

### Custom Validation with Logging

```cpp
GripperRuntimeValidator validator("DH56");

// Set custom tolerances
validator.set_position_tolerance(0.02f);  // 2% tolerance
validator.set_force_tolerance(0.05f);     // 5% tolerance

// Validate with detailed logging to stdout
std::cout << "Starting validation...\n";
auto result = validator.validate_responsiveness(
    3000,  // 3 second timeout
    &std::cout  // Log stream
);

// Check result
if (result.is_valid()) {
    std::cout << "✓ All checks passed\n";
    std::cout << "  Latency: " << result.latency_ms.count() << " ms\n";
    std::cout << "  Status: " << 
        GripperRuntimeValidator::format_status_(result.received_status.value()) << "\n";
} else {
    std::cerr << "✗ Validation failed: " << result.message << "\n";
    std::cerr << "  Details: " << result.diagnostics << "\n";
}
```

### Sequential Multi-Device Validation

```cpp
std::vector<std::string> devices = {"DH56", "CGC-80"};
std::map<std::string, ValidationResult> all_results;

for (const auto& device_id : devices) {
    GripperRuntimeValidator validator(device_id);
    auto result = validator.validate_responsiveness(2000);
    all_results[device_id] = result;
    
    std::cout << device_id << ": " 
              << (result.is_valid() ? "✓ PASS" : "✗ FAIL") << "\n";
    
    if (!result.is_valid()) {
        std::cerr << "  " << result.message << "\n";
    }
}
```

### Sequence of Commands Within Single Device

```cpp
GripperRuntimeValidator validator("DH56");

// Test motion range: 0 → 0.5 → 1.0 → 0.5 → 0
std::vector<std::tuple<float, float, float, GripperMode>> commands = {
    {0.0f, 0.3f, 0.2f, GripperMode::MODE_POSITION},  // Open
    {0.5f, 0.5f, 0.3f, GripperMode::MODE_POSITION},  // Mid
    {1.0f, 0.7f, 0.4f, GripperMode::MODE_POSITION},  // Close
    {0.5f, 0.6f, 0.3f, GripperMode::MODE_FORCE},     // Force mode
};

auto results = validator.validate_sequence(commands, 2000);

int passed = 0, failed = 0;
for (size_t i = 0; i < results.size(); ++i) {
    if (results[i].is_valid()) {
        std::cout << "[" << (i+1) << "] ✓ PASS\n";
        ++passed;
    } else {
        std::cout << "[" << (i+1) << "] ✗ FAIL: " 
                  << results[i].message << "\n";
        ++failed;
    }
}
std::cout << "Total: " << passed << " passed, " << failed << " failed\n";
```

## Result Structure

The `ValidationResult` struct contains:

```cpp
struct ValidationResult {
    ValidationStatus status;           // SUCCESS, TIMEOUT, MISMATCH, etc.
    std::string device_id;             // Device being validated
    std::string message;               // Human-readable summary
    std::string diagnostics;           // Detailed diagnostic info
    
    // Command details
    float commanded_position;
    float commanded_force;
    float commanded_speed;
    qnc::gripper::GripperMode commanded_mode;
    
    // State feedback (if received)
    std::optional<float> received_position;
    std::optional<float> received_force;
    std::optional<float> received_speed;
    std::optional<qnc::gripper::GripperStatus> received_status;
    
    // Timing
    std::chrono::milliseconds latency_ms;
    
    bool is_valid() const;             // Returns true if status == SUCCESS
    std::string status_string() const; // "SUCCESS", "TIMEOUT", etc.
};
```

## Validation Status Codes

| Status | Meaning | Typical Cause |
|--------|---------|---------------|
| `SUCCESS` | Command issued, state received, all parameters validated | Normal operation |
| `TIMEOUT` | No state feedback within timeout period | Hardware unresponsive, DDS network issue |
| `STATE_MISMATCH` | State received but parameters don't match | Gripper didn't execute command correctly |
| `COMMUNICATION_ERROR` | DDS initialization failed | DDS domain/participant/topic creation error |
| `INVALID_DEVICE_ID` | Device not found or invalid | Device ID not connected, routing issue |
| `UNKNOWN_ERROR` | Unexpected error | Unforeseen condition |

## Tolerances (Defaults)

Validation uses **normalized value tolerances** (0.0–1.0 range):

| Parameter | Default Tolerance | Reasoning |
|-----------|-------------------|-----------|
| Position | 0.05 (5%) | Gripper mechanical variance, rounding |
| Force | 0.10 (10%) | Force sensor drift, device variation |
| Speed | 0.10 (10%) | Speed control variations |

These are lenient by design. To make validation stricter (for production):

```cpp
validator.set_position_tolerance(0.02f);  // 2% (strict)
validator.set_force_tolerance(0.05f);     // 5% (strict)
validator.set_speed_tolerance(0.05f);     // 5% (strict)
```

## Integration Examples

### 1. Startup Health Check

Add to `qnc_bridge` startup script:

```bash
#!/bin/bash
set -e

# Validate hardware before starting main bridge
echo "Performing hardware validation..."
./gripper_validation_test --device DH56 --timeout 3000
if [ $? -ne 0 ]; then
    echo "ERROR: Hardware validation failed. Not starting bridge."
    exit 1
fi

# Hardware OK, start bridge
./qnc_bridge
```

### 2. Continuous Integration

```bash
# ci-test.sh
./gripper_validation_test --sequence --quiet
exit $?
```

### 3. Multi-Device Robot Validation

```cpp
// In your robot initialization code
std::vector<std::string> grippers = {"AG160", "CGC-80", "DH56"};
std::vector<ValidationResult> results;

for (const auto& dev : grippers) {
    std::cout << "Validating " << dev << "... ";
    auto r = validate_gripper_device(dev, 3000, false);
    results.push_back(r);
    std::cout << (r.is_valid() ? "✓\n" : "✗\n");
}

// Fail if any gripper is not responsive
for (const auto& r : results) {
    if (!r.is_valid()) {
        throw std::runtime_error("Gripper " + r.device_id + " failed validation");
    }
}
```

## Troubleshooting

### Validation Always Times Out

**Symptom:** No state feedback even after increasing timeout

**Causes & Fixes:**

1. **Bridge not running**
   ```bash
   ps aux | grep qnc_bridge
   # Start it: ./qnc_bridge
   ```

2. **Wrong DDS domain ID**
   ```bash
   # Check bridge domain (default: 0)
   # Validator also defaults to domain 0
   # If bridge uses domain 1:
   export QNC_DOMAIN_ID=1
   ./qnc_bridge
   ```

3. **Gripper hardware offline**
   ```bash
   # Check serial connection
   ls -la /dev/gripper_*
   # Verify with modbus_rtu_slave_test if available
   ```

4. **DDS network unreachable**
   ```bash
   # Check DDS network via environment vars (FastDDS)
   export ROS_DOMAIN_ID=0
   ./gripper_validation_test
   ```

### State Received But Parameters Don't Match

**Symptom:** STATE_MISMATCH validation failure

**Causes & Fixes:**

1. **Tolerance too strict**
   ```bash
   ./gripper_validation_test --pos-tol 0.1 --force-tol 0.15
   ```

2. **Device descriptor mapping is incorrect**
   - Check `QNC_GRIPPER_DESCRIPTOR` env var points to correct JSON
   - Verify position/force/speed scaling in descriptor

3. **Gripper not responding to commands**
   - Check RS485 wiring
   - Verify Modbus slave address in descriptor matches hardware

### Latency Too High

**Symptom:** Command latency > 500ms

**Causes:**

1. System under high load — retries usually succeed
2. Serial port baud rate too slow
3. Gripper hardware slow to respond (normal for some devices)

## Best Practices

1. **Always validate before deployment**
   ```bash
   ./gripper_validation_test --sequence
   ```

2. **Use reasonable timeouts**: 2000–5000 ms for typical hardware

3. **Keep default tolerances** unless you have specific requirements

4. **Log validation results** for diagnostics
   ```bash
   ./gripper_validation_test 2>&1 | tee validation-$(date +%s).log
   ```

5. **Test after hardware changes**
   - After replacing a gripper
   - After re-tuning Modbus parameters
   - After changing descriptor mappings

6. **Extend for your use case**
   - Add custom validation logic in C++
   - Create test sequences for your specific application
   - Integrate with your CI/CD pipeline
