# Nucleo Firmware — Integration & Wiring Guide

## Hardware Stack

```
 STEVAL-IOM001V1  (L6360 IO-Link master PHY shield)
       ↕  Arduino + morpho stacking connectors (no loose wires to PHY)
 NUCLEO-F746ZG    (STM32F746ZG)
       ↕  USB cable
 PC / CubeIDE     (for programming)
```

```
 NUCLEO-F746ZG  ──  SPI1 slave (PA4/PA5/PA6/PB5)  ──  RPi5 SPI0
```

```
 STEVAL-IOM001V1  ──  M12 port 1 connector  ──  Zimmer LWR50L-02 (M12 cable)
```

---

## Power Connections (STEVAL-IOM001V1)

| STEVAL terminal | Connect to | Notes |
|---|---|---|
| J1 VIN+ | 24 V DC PSU + | Powers L+ output on M12 port |
| J1 GND  | 24 V DC PSU − | Common ground |

The Nucleo is powered separately via its USB connector (5 V from PC / USB charger).
Do **not** connect 24 V to the Nucleo directly.

---

## Arduino Connector Pin Mapping (STEVAL port 1 → Nucleo)

These signals are automatically routed when you stack the shield — no jumper wires needed.

| Arduino pin | STM32 pin | Signal | Direction | Function |
|---|---|---|---|---|
| D1 | PA2 | USART2_TX / INCQ | Nucleo → STEVAL | IO-Link data TX to L6360 |
| D0 | PA3 | USART2_RX / OUTCQ | STEVAL → Nucleo | IO-Link data RX from L6360 |
| D8 | PA9 | ENCQ (GPIO OUT) | Nucleo → STEVAL | CQ driver enable, HIGH=TX, LOW=RX |
| D9 | PC7 | ENL+ (GPIO OUT) | Nucleo → STEVAL | L+ 24V supply enable, HIGH=on |
| D2 | PA10 | IRQ (GPIO IN) | STEVAL → Nucleo | L6360 fault output, active LOW |

## RPi5 ↔ Nucleo SPI Wiring (unchanged from previous setup)

| RPi5 physical pin | RPi5 GPIO | Nucleo pin | Signal |
|---|---|---|---|
| 19 | GPIO10 / MOSI | PB5 / SPI1_MOSI | MOSI |
| 21 | GPIO9  / MISO | PA6 / SPI1_MISO | MISO |
| 23 | GPIO11 / SCLK | PA5 / SPI1_SCK  | SCK  |
| 24 | GPIO8  / CE0  | PA4 / SPI1_NSS  | CS (active LOW) |
| 25 | GND           | GND             | Common ground |
| 1  | 3V3           | 3V3             | (optional, for reference level only) |

> **Note**: The RPi5 3.3 V and Nucleo 3.3 V must share a common ground.

## CubeMX / CubeIDE Setup (if regenerating with CubeMX)

When regenerating the CubeMX project:

1. **USART2** — Mode: Asynchronous, Baud: 38400, Word: 8 bits, Parity: None, Stop: 1  
   PA2 = USART2_TX (AF7), PA3 = USART2_RX (AF7)

2. **GPIO outputs**  
   - PA9 → GPIO_Output, label `ENCQ`, default LOW  
   - PC7 → GPIO_Output, label `ENL_PLUS`, default LOW  

3. **GPIO input**  
   - PA10 → GPIO_Input, pull-up, label `IOLINK_IRQ`

4. **SPI1** — Slave Full-Duplex, 8-bit, CPOL=0 CPHA=0, HW NSS  
   PA4=NSS, PA5=SCK, PA6=MISO, PB5=MOSI (AF5)

5. **After generating code**, add to `main.c`:

   ```c
   /* USER CODE BEGIN Includes */
   #include "app.h"
   /* USER CODE END Includes */

   /* USER CODE BEGIN 2 */
   app_init(&hspi1);
   /* USER CODE END 2 */

   /* USER CODE BEGIN WHILE */
   app_run();
   /* USER CODE END WHILE */
   ```

> If building with the standalone **Makefile** (no CubeMX), `main.c` in this repo  
> includes all required `MX_xxx_Init()` functions and MSP callbacks — no changes needed.

---

## What `app.c` Does

- `app_init()` — calls `iolink_init()` and arms the first SPI slave transfer
- `app_run()` — processes one pending SPI command per call:
  - `CMD_GRIP (0x01)` → runs `iolink_cycle(GRIP)` → sets `spi_tx_byte = RSP_GRIPPING`
  - `CMD_RELEASE (0x02)` → runs `iolink_cycle(RELEASE)` → sets `spi_tx_byte = RSP_RELEASING`
  - `CMD_STATUS (0x03)` → reads PDin from device → returns GRIPPING / RELEASING / IDLE
- `HAL_SPI_TxRxCpltCallback()` — defined in `app.c`; sets `new_cmd_flag` only

## Files Tracked in Git

| File | Purpose |
|---|---|
| `Core/Src/app.c` | Application logic (SPI command dispatcher) |

## Files NOT Tracked in Git (CubeMX-owned, for CubeIDE builds)

When using CubeIDE, CubeMX regenerates and **owns** these files (do not edit manually):

- `Core/Src/main.c` — use the 3 integration snippets above instead
- `Core/Src/stm32f7xx_hal_msp.c` — generated GPIO/clock init for each peripheral
- `Core/Src/stm32f7xx_it.c` — ISR stubs
- `Core/Src/system_stm32f7xx.c` — system clock setup
